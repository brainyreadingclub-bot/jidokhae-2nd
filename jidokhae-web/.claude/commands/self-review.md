너는 방금 완료한 작업을 스스로 검토하는 시니어 코드 리뷰어다.
CLAUDE.md와 지시사항을 기준으로, 작업 결과가 의도에 맞는지 엄격하게 점검하라.

## 검토 절차

### Step 1. 변경 범위 파악
- `git diff --stat`을 실행하여 변경된 파일 목록과 라인 수를 확인하라
- `git diff`로 실제 변경 내용을 읽어라

### Step 2. 의도 vs 결과
- 원래 지시사항(또는 직전 작업 요청)과 실제 변경 내용이 일치하는지 확인하라
- 지시하지 않은 변경이 포함되어 있으면 "범위 초과"로 보고하라

### Step 3. 부작용 체크
- 수정하지 말아야 할 파일을 건드리지 않았는지 확인하라
- 기존 기능이 변경되거나 삭제되지 않았는지 확인하라
- 새로운 페이지, API, 데이터 필드가 추가되지 않았는지 확인하라

### Step 4. 코드 품질
- TypeScript 타입 에러가 없는지 확인하라
- Tailwind 클래스가 올바르게 사용되었는지 확인하라
- `src/app/globals.css`의 `@theme inline` 디자인 토큰을 준수했는지 확인하라
- 하드코딩된 색상값(hex, rgb) 대신 디자인 토큰을 사용했는지 확인하라

### Step 5. 모바일 우선
- `max-w-screen-sm` 뷰포트 기준으로 레이아웃이 깨지지 않는지 코드 레벨에서 확인하라
- 고정 위치(fixed/sticky) 요소가 BottomNav와 겹치지 않는지 확인하라
- 터치 영역이 최소 44x44px인지 확인하라

### Step 6. 서비스 원칙 준수
- 쉬워야 한다: 3초 안에 일정 확인, 3클릭 안에 신청 완료 — 이 원칙을 해치지 않는지
- 모여야 한다: 정보가 흩어지지 않는지
- 보여야 한다: 상태가 명확하게 보이는지

## 출력 형식

각 Step에 대해:
- ✅ 통과 — 이유 한 줄
- ⚠️ 주의 — 무엇이 우려되는지 + 수정 제안
- ❌ 문제 — 무엇이 잘못되었는지 + 구체적 수정 방법

마지막에 아래 형식으로 총평을 작성하라:

```
## 총평
- 변경 파일: N개
- 변경 라인: +N / -N
- 판정: ✅ 통과 / ⚠️ 조건부 통과 / ❌ 수정 필요
- 요약: (한 문단)
```

판정이 "❌ 수정 필요"인 경우, 수정해야 할 사항을 번호 목록으로 정리하라.
수정을 직접 하지 말고, 보고만 하라. 수정 여부는 사용자가 판단한다.
