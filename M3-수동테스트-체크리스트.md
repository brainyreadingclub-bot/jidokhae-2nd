> M3 (모임 일정 조회 + 운영자 CRUD) 구현 완료 후 수동 검증을 위한 체크리스트.
> 테스트 전에 아래 "임시 데이터 준비" 섹션의 데이터를 먼저 생성할 것.

## 1. 임시 데이터 준비

### 1-1. 계정 준비

| # | 계정 | 역할 | 준비 방법 |
|---|------|------|-----------|
| A1 | member 계정 | `role = 'member'` | 카카오 로그인 (기본값이 member이므로 그대로 사용) |
| A2 | admin 계정 | `role = 'admin'` | 카카오 로그인 후 Supabase 대시보드에서 `UPDATE profiles SET role = 'admin' WHERE id = '해당UUID'` |

### 1-2. 모임 데이터 (admin 계정으로 생성)

> **모든 날짜는 테스트 당일 기준으로 조정할 것.** 아래는 예시 (테스트일 = 2026-03-12 가정).

| # | 모임명 | 날짜 | 시간 | 장소 | 정원 | 참가비 | 용도 |
|---|--------|------|------|------|:----:|-------:|------|
| M1 | 오늘 독서모임 | **오늘** (3/12) | 19:00 | 경주 카페A | 14 | 10,000 | 오늘 모임 표시 확인, 버튼 분기 테스트 |
| M2 | 내일 독서모임 | **내일** (3/13) | 14:00 | 포항 카페B | 14 | 15,000 | 미래 모임, 신청하기 버튼 테스트 |
| M3 | 다음주 독서모임 | **7일 후** (3/19) | 10:00 | 경주 카페C | 2 | 10,000 | **정원 2명 → 마감 테스트용** |
| M4 | 수정/삭제 테스트 모임 | **5일 후** (3/17) | 18:00 | 포항 카페D | 10 | 20,000 | CRUD 수정/삭제 테스트용 |
| M5 | 과거 독서모임 | **어제** (3/11) | 19:00 | 경주 카페E | 14 | 10,000 | 과거 모임 필터링, "참여 완료" 버튼 테스트 |

### 1-3. 신청(registration) 데이터 (Supabase SQL Editor에서 생성)

> M3 단계에서는 결제 파이프라인이 없으므로, registrations 레코드를 **SQL로 직접 INSERT** 해야 합니다.
> `{member_user_id}` = A1(member 계정)의 profiles.id UUID 값

```sql
-- ⚠️ 아래 UUID들을 실제 값으로 교체할 것

-- (1) M3 모임 (정원 2) — confirmed 2건 → 마감 상태 만들기
--     가짜 유저 2명의 신청이 필요 (또는 admin + member)
INSERT INTO registrations (user_id, meeting_id, status, payment_id, paid_amount)
VALUES
  ('{admin_user_id}', '{M3_meeting_id}', 'confirmed', 'test_pay_001', 10000),
  ('{member_user_id}', '{M3_meeting_id}', 'confirmed', 'test_pay_002', 10000);

-- (2) M5 과거 모임 — member가 confirmed 신청 보유 → "참여 완료" 테스트
INSERT INTO registrations (user_id, meeting_id, status, payment_id, paid_amount)
VALUES
  ('{member_user_id}', '{M5_meeting_id}', 'confirmed', 'test_pay_003', 10000);

-- (3) M2 내일 모임 — member가 confirmed 신청 보유 → "신청완료" 뱃지 + "취소하기" 버튼
INSERT INTO registrations (user_id, meeting_id, status, payment_id, paid_amount)
VALUES
  ('{member_user_id}', '{M2_meeting_id}', 'confirmed', 'test_pay_004', 15000);

-- (4) M1 오늘 모임 — member의 cancelled 레코드만 존재 → 미신청 취급 확인
INSERT INTO registrations (user_id, meeting_id, status, cancel_type, payment_id, paid_amount, refunded_amount, cancelled_at)
VALUES
  ('{member_user_id}', '{M1_meeting_id}', 'cancelled', 'user_cancelled', 'test_pay_005', 10000, 10000, now());
```

### 1-4. 상태별 모임 데이터 (SQL로 직접 생성)

> deleting/deleted 상태 모임은 admin 폼으로 생성할 수 없으므로 SQL로 만들기.

```sql
-- deleting 상태 모임 (회원 목록에 안 보여야 함)
INSERT INTO meetings (title, date, time, location, capacity, fee, status)
VALUES ('환불중 모임', '2026-03-20', '19:00', '경주 카페F', 14, 10000, 'deleting');

-- deleted 상태 모임 (누구에게도 안 보여야 함 — 운영자 목록에서도)
INSERT INTO meetings (title, date, time, location, capacity, fee, status)
VALUES ('삭제된 모임', '2026-03-01', '19:00', '포항 카페G', 14, 10000, 'deleted');
```
