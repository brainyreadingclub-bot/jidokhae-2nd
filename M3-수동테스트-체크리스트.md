# M3 수동 테스트 체크리스트

M3 (모임 일정 조회 + 운영자 CRUD) 구현 완료 후 수동 검증용.

---

## STEP 1. 계정 준비

두 개의 카카오 계정이 필요합니다.

| 계정 | 역할 | 준비 방법 |
|------|------|-----------|
| **A1** (member) | `member` | 카카오 로그인하면 기본값이 member |
| **A2** (admin) | `admin` | 카카오 로그인 후, Supabase SQL Editor에서 아래 실행 |

```sql
-- A2 계정을 admin으로 변경
UPDATE profiles SET role = 'admin' WHERE id = '여기에_A2의_UUID';
```

## STEP 2. 모임 데이터 생성 (admin 계정 A2로)

admin 계정으로 로그인 후, `/admin/meetings/new`에서 아래 5개 모임을 직접 생성합니다.

> 날짜는 **테스트 당일 기준**으로 조정하세요.

| # | 모임명 | 날짜 | 시간 | 장소 | 정원 | 참가비 |
|---|--------|------|------|------|:----:|-------:|
| M1 | 오늘 독서모임 | 오늘 | 19:00 | 경주 카페A | 14 | 10,000 |
| M2 | 내일 독서모임 | 내일 | 14:00 | 포항 카페B | 14 | 15,000 |
| M3 | 다음주 독서모임 | 7일 후 | 10:00 | 경주 카페C | **2** | 10,000 |
| M4 | 수정삭제 테스트모임 | 5일 후 | 18:00 | 포항 카페D | 10 | 20,000 |
| M5 | 과거 독서모임 | **어제** | 19:00 | 경주 카페E | 14 | 10,000 |

각 모임의 용도:
- **M1** — 오늘 모임 표시 확인, cancelled 유저의 버튼 분기
- **M2** — 미래 모임, "신청완료" 뱃지 + "취소하기" 버튼
- **M3** — 정원 2명으로 마감 테스트
- **M4** — 수정/삭제 CRUD 테스트
- **M5** — 과거 모임 필터링, "참여 완료" 버튼 (목록에는 안 보이고 URL 직접 접근으로 테스트)

## STEP 3. 신청(registration) 데이터 INSERT

M3 단계에서는 결제가 없으므로, **Supabase SQL Editor에서 직접 INSERT**합니다.

먼저 필요한 UUID를 확인하세요:
```sql
-- member(A1)의 UUID 확인
SELECT id, nickname, role FROM profiles WHERE role = 'member';

-- admin(A2)의 UUID 확인
SELECT id, nickname, role FROM profiles WHERE role = 'admin';

-- 모임 UUID 확인
SELECT id, title, date FROM meetings ORDER BY date;
```

확인한 UUID로 아래 `{...}` 부분을 교체 후 실행:

```sql
-- (1) M3 마감 만들기: 정원 2에 confirmed 2건
INSERT INTO registrations (user_id, meeting_id, status, payment_id, paid_amount)
VALUES
  ('{admin_user_id}',  '{M3_meeting_id}', 'confirmed', 'test_pay_001', 10000),
  ('{member_user_id}', '{M3_meeting_id}', 'confirmed', 'test_pay_002', 10000);

-- (2) M5 과거 모임: member가 confirmed → "참여 완료" 테스트
INSERT INTO registrations (user_id, meeting_id, status, payment_id, paid_amount)
VALUES
  ('{member_user_id}', '{M5_meeting_id}', 'confirmed', 'test_pay_003', 10000);

-- (3) M2 내일 모임: member가 confirmed → "신청완료" 뱃지 + "취소하기" 버튼
INSERT INTO registrations (user_id, meeting_id, status, payment_id, paid_amount)
VALUES
  ('{member_user_id}', '{M2_meeting_id}', 'confirmed', 'test_pay_004', 15000);

-- (4) M1 오늘 모임: member의 cancelled만 존재 → 미신청 취급 확인
INSERT INTO registrations
  (user_id, meeting_id, status, cancel_type, payment_id, paid_amount, refunded_amount, cancelled_at)
VALUES
  ('{member_user_id}', '{M1_meeting_id}', 'cancelled', 'user_cancelled', 'test_pay_005', 10000, 10000, now());
```

## STEP 4. 상태별 모임 데이터 (SQL로 직접 생성)

`deleting`/`deleted` 상태는 admin 폼으로 만들 수 없으므로 SQL로 생성합니다.

```sql
-- deleting 상태 (회원 목록에 안 보여야 함, admin 목록에는 보여야 함)
INSERT INTO meetings (title, date, time, location, capacity, fee, status)
VALUES ('환불중 모임', '2026-03-20', '19:00', '경주 카페F', 14, 10000, 'deleting');

-- deleted 상태 (누구에게도 안 보여야 함)
INSERT INTO meetings (title, date, time, location, capacity, fee, status)
VALUES ('삭제된 모임', '2026-03-01', '19:00', '포항 카페G', 14, 10000, 'deleted');
```

> 위 날짜도 테스트 당일 기준으로 조정하세요. `환불중 모임`은 미래 날짜, `삭제된 모임`은 과거 날짜.

## STEP 6. 테스트 후 정리

```sql

DELETE FROM registrations WHERE payment_id LIKE 'test_pay_%';

-- 2. 상태 테스트용 모임 삭제
DELETE FROM meetings WHERE title IN ('환불중 모임', '삭제된 모임');

```
