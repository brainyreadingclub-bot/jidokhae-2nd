---
name: 사후 상태 잔재 점검
description: 운영자 UI 변경 시 "이 상태가 만들어진 후 일어나는 잔재 케이스" 사전 점검 필수 — 모임 삭제 후 환불 대기, 회원 잠수, 토글 누락 등
type: feedback
---

운영자가 보는 신청자 표·대시보드 카운트 등을 만들거나 고칠 때, **"이 상태가 만들어진 직후"만이 아니라 "이 상태가 그대로 며칠 남은 후"** 일어나는 잔재 케이스를 사전에 점검해야 한다.

**Why:**
M7 Step 2.6에서 RefundToggle을 "환불 대기 → 송금 후 토글" 흐름만 가정하고 설계했다. 모임 삭제 후 계좌이체 confirmed 회원이 자동 환불 대기로 남는 케이스를 빠뜨렸고, deleted 모임은 admin 상세 페이지가 `notFound()` 분기라 RefundToggle 사용 경로 자체가 차단됨을 사용자가 "내가 모임 자체를 삭제한 거 같은데" 짚은 후에야 발견. 결국 사용자가 SQL로 직접 처리해야 했고, 구조적 한계로 다음 세션 후보 작업으로 밀림.

또한 pending_transfer 상태에서 회원이 잠수타면 정원이 영구 점유되는 자동 만료 부재도 사용자가 "이 숫자는 쌓이게 돼있는 것 같아" 짚어준 후에야 인지함.

**How to apply:**
운영자 화면 카운트·라벨·토글을 신설/수정하기 전에 다음 4가지 잔재 시나리오를 plan에 명시 점검:
1. **상태가 만들어진 후 며칠 잠수**: 자동 만료 메커니즘이 있나? 영구 누적되나?
2. **상위 엔티티 삭제 후 잔재**: 모임 삭제 후 그 모임의 신청들이 어떻게 표시되고 처리 가능한가? (deleted 모임의 admin 상세 페이지 접근 여부 확인)
3. **부분 처리 누락**: 운영자가 토글을 깜빡 누락하면 시스템과 실제 송금/입금이 어긋나는가? 정정 경로(unmark, OFF 토글)가 있나?
4. **회원 측 후속 행동**: 회원이 카톡으로 연락 → 운영자가 어떤 정보(전화번호·계좌 등)로 회원 식별하나? 시스템에 그 정보가 보이나?

이 점검을 plan의 "사용자 흐름 N각도 점검"에서 별도 각도로 박을 것.
