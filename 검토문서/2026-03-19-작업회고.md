## 잘한 점

### 1. 지시서 맹목 수행하지 않고 검토함

지시서의 "수정 1: WelcomeScreen에 닉네임 표시"가 **이미 구현되어 있음**을 발견하고 스킵했다.
맹목적으로 따랐으면 중복 코드를 만들었을 것.

### 2. 모바일 스크롤 문제 사전 발견

13개 지역 버튼 + 폼 4개가 360x600 뷰포트에서 넘치는 문제를 코드 리뷰 단계에서 발견.
`fixed + justify-center` → `overflow-y-auto + flex min-h-full` 패턴으로 수정.
배포 후 사용자가 CTA 버튼을 못 누르는 상황을 방지했다.

### 3. WelcomeScreen CTA 동작 변경의 부작용을 분석함

`router.push('/')` 제거 시 에러 동작이 바뀌는 점을 파악:
- 기존: API 실패해도 메인으로 이동 (진입 차단 안 함)
- 변경 후: API 실패 시 웰컴 스크린 재표시 (재시도 가능)

프로필 설정 플로우에서는 변경 후 동작이 더 적절하다고 판단하고 진행.

## 아쉬운 점 / 교훈

### 1. 지역 선택 스펙이 3번 바뀜

경주/포항/대구 3개 → 13개 도시 → 복수 선택. DB 컬럼 타입이 TEXT → TEXT(CHECK 변경) → TEXT[]로 3번 바뀌었다.
사용자가 이미 첫 번째 마이그레이션을 실행한 상태여서, 교체용 SQL을 별도로 안내해야 했다.

**교훈:** 사용자 입력 관련 스펙(선택지 종류, 단일/복수)은 구현 전에 확인하는 게 좋다.
지시서에 "경주/포항/대구 버튼 3개"라고 되어 있어도, 실제 서비스 상황(다양한 지역 회원)을 한 번 더 물어봤으면 한 번에 처리할 수 있었다.

### 2. DB 마이그레이션을 코드보다 먼저 배포함

사용자가 첫 마이그레이션(3개 지역 CHECK)을 바로 실행했는데, 이후 코드에서 지역 목록이 바뀌면서 CHECK 제약 불일치가 발생할 수 있었다.

**교훈:** 마이그레이션 SQL은 코드 변경이 확정된 후에 안내하는 것이 안전하다.
특히 CHECK 제약처럼 코드와 동기화가 필요한 것은 더더욱.

### 3. 브랜치 전략이 중간에 흐려짐

`feature/profile-setup` 브랜치를 만들었지만, 결국 빠르게 main에 머지+푸시했다.
리뷰 없이 main에 직접 머지하는 패턴이 반복되면 실수 시 롤백이 어려워진다.
