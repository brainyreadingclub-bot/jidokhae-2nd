# JIDOKHAE MVP 검토 결과: 문제와 해결방안

> 마일스톤, 워크패키지, 시나리오 문서를 결제 서비스 출시 관점에서 검토한 결과입니다.
> 기준: "250명 독서모임이 카카오톡 인앱 브라우저에서 접속하여 결제하는 서비스"

---

## 문제 1. 결제했는데 신청이 안 되는 경우가 있다 (Webhook 누락)

### 문제 상황

현재 설계된 결제 흐름:

```
회원이 결제 완료 (돈 빠져나감)
    ↓
회원 폰이 우리 서버에 "결제 끝났어, 확인해줘" 요청을 보냄
    ↓
서버가 포트원에 확인 → DB에 신청 기록 저장
    ↓
"신청 완료!"
```

회원이 결제를 완료해서 돈은 빠져나갔는데, 그 다음 단계인 "서버에 확인 요청"이 전달되지 않는 경우가 존재한다.

발생 원인: 결제 완료 직후 카카오톡으로 돌아가거나, 인터넷이 잠깐 끊기거나, 브라우저가 꺼지는 경우. 특히 카카오톡 인앱 브라우저에서 빈번하게 발생할 수 있다.

결과: 돈은 나갔는데 신청 목록에 안 뜨는 상태. 현재 문서(Scenario 4-1-09)에서는 이를 "MVP 한계"로 남겨두고 포트원 콘솔에서 운영자가 수동 확인하도록 되어있다.

### 해결방안: 포트원 Webhook 추가

Webhook을 추가하면 포트원이 직접 우리 서버에 "이 결제 됐어"라고 알려주는 백업 경로가 생긴다. 회원 폰이 꺼져도 상관없이 신청이 자동 처리된다.

```
[정상 경로]  회원 폰 → 우리 서버 → 포트원 확인 → 등록
[백업 경로]  포트원 → 우리 서버 → 이미 등록됐나? → 안됐으면 등록
```

### 반영 위치

- **WP4-1 산출물 추가**: `/api/webhooks/portone` API Route. 포트원 결제 완료 Webhook 수신 → 미등록 결제 건 자동 처리. 이미 등록된 건은 무시 (문제 3의 중복 방어와 연동)
- **Scenario 4-1-09 수정**: 기존 "MVP 한계 — 포트원 콘솔에서 운영자 수동 확인 필요" → "Webhook으로 자동 복구. 프론트 콜백 실패 시에도 포트원 Webhook이 서버에 직접 결제 완료를 통지하여 등록 처리"
- **기술 스택 §2-4 위험성 갱신**: line 118 "결제 Webhook 실패 시 → 서버 사이드 결제 검증으로 해결" 문구를 Webhook 백업 경로 추가에 맞게 수정. 클라이언트 콜백 + Webhook 이중 경로 구조로 설명 변경

### 복잡도 평가

WP4-1에 이미 "결제 검증 + 환불 기능을 재사용 가능한 모듈로 구현"이 설계되어 있으므로, Webhook은 이 모듈을 재사용하는 API Route 하나(코드 50~80줄) + 포트원 대시보드 URL 등록(5분)으로 구현 가능하다. 나중에 추가하는 것보다 처음부터 넣는 게 안전하다.

## 문제 2. 카카오톡에서 결제 후 돌아오는 처리가 없다 (redirect 복귀 로직 누락)

### 문제 상황

지독해 회원의 대부분은 카카오톡 오픈채팅방에서 링크를 클릭하여 서비스에 접속한다. 이때 카카오톡 앱 안에 내장된 미니 브라우저("인앱 브라우저")에서 웹페이지가 열린다.

결제창을 여는 방식에는 두 가지가 있다:
- **popup 방식**: 결제창이 팝업으로 뜨고, 결제 후 원래 페이지에서 콜백 함수가 실행됨
- **redirect 방식**: 현재 페이지가 결제 페이지로 이동하고, 결제 후 지정된 URL(returnUrl)로 돌아옴

카카오톡 인앱 브라우저에서는 popup이 차단되거나 불안정하여 redirect 방식이 사용될 가능성이 높다. 그런데 현재 WP4-1/4-2에는 popup 콜백 방식만 고려되어 있고, **redirect로 돌아왔을 때 결제 결과를 읽어서 검증 API를 호출하는 "복귀 페이지" 로직이 없다.**

결과: 카카오톡 인앱에서 결제를 완료해도 돌아올 페이지가 없어서 결제 확정 처리가 안 될 수 있다.

### 해결방안: 결제 완료 후 redirect 복귀 페이지 + 결과 처리 로직 추가

결제 완료 후 돌아올 페이지(returnUrl)를 만들고, 해당 페이지에서 결제 결과를 읽어 검증 API(`/api/registrations/confirm`)를 호출하는 로직을 추가한다. popup 콜백과 redirect 복귀 두 경로 모두 동작해야 한다.

### 반영 위치

- **WP4-1 산출물 추가**: "결제 완료 redirect 복귀 페이지(returnUrl) 처리. popup 콜백과 redirect 복귀 두 경로 모두 지원"
- **WP4-2 산출물 추가**: "redirect 복귀 시에도 결제 검증 → 확정 화면 → 목록 복귀 흐름이 동일하게 동작"
- **기술 스택 §4-2 흐름 2**: ①번 단계에 결제창 모드(popup/redirect) 명시, ②번 단계에 popup 콜백 + redirect returnUrl 복귀 두 경로 병기

## 문제 3. 같은 사람이 같은 모임에 두 번 등록될 수 있다 (중복 등록 방어 누락)

### 문제 상황

기술 스택 문서의 `confirm_registration` DB Function 의사코드:

```
1. SELECT ... FROM meetings WHERE id = p_meeting_id FOR UPDATE (행 잠금)
2. SELECT COUNT(*) FROM registrations WHERE meeting_id AND status = 'confirmed'
3. IF count < capacity → INSERT
```

이 로직에는 "이 사용자가 이 모임에 이미 confirmed 상태로 등록되어 있는지" 확인하는 단계가 없다. Scenario 4-1-08과 WP4-1 엣지 케이스에서는 중복 방어를 요구하지만, 실제 구현 스펙(의사코드)에 반영되지 않은 상태이다.

결과: 같은 사람이 같은 모임에 confirmed 신청이 2개 생길 수 있고, 정원 계산이 틀어진다. 문제 1의 Webhook 자동 복구도 이 중복 방어 로직에 의존한다.

### 해결방안: DB Function 의사코드에 중복 체크 추가

```
1. SELECT ... FROM meetings WHERE id = p_meeting_id FOR UPDATE
1.5. SELECT COUNT(*) FROM registrations 
     WHERE user_id = p_user_id AND meeting_id = p_meeting_id AND status = 'confirmed'
     → count > 0이면 'already_registered' 반환
2. SELECT COUNT(*) FROM registrations WHERE meeting_id AND status = 'confirmed'
3. IF count < capacity → INSERT
```

### 반영 위치

- **WP1-2 산출물 6번** (`confirm_registration`): 의사코드에 1.5단계 추가
- **기술 스택 §4-2 흐름 2**: DB Function 의사코드 동일 수정
- **Scenario 1-2-14/15 추가**: 중복 방어 테스트 시나리오 (1-2-12/13은 이미 정원 체크 시나리오로 사용 중)

## 문제 4. 버튼을 빠르게 두 번 누르면 이중 결제/이중 취소가 된다 (중복 클릭 방어 누락)

### 문제 상황

WP4-2(신청 UX) 산출물 5개, WP5-1(취소) 산출물 10개 어디에도 "버튼 비활성화" 또는 "로딩 상태"에 대한 언급이 없다.

"신청하기" 버튼을 빠르게 두 번 누르면 결제창이 두 개 열릴 수 있다. "취소하기" 확정 버튼도 마찬가지로 환불 API가 두 번 호출될 수 있다.

결과: 이중 결제 또는 이중 환불 발생 가능.

### 해결방안: UI 버튼 비활성화 + API 멱등성 보장

**프론트엔드**: 클릭 시 버튼을 즉시 비활성화하고 로딩 인디케이터를 표시한다. 결제 완료/실패 또는 API 응답이 돌아올 때까지 유지한다.

**백엔드 (취소 API)**: UI 방어만으로는 불완전하다. 취소 API Route(`/api/registrations/cancel`)에서 트랜잭션 시작 시 `registrations.status = 'confirmed'`를 재확인하고, 이미 `cancelled`이면 환불 API를 호출하지 않고 즉시 반환한다 (멱등성 보장). 이는 네트워크 재시도, 브라우저 새로고침 등 UI 방어를 우회하는 경우에도 이중 환불을 방지한다.

> 참고: 결제의 경우 포트원 V2 SDK가 내부적으로 중복 결제창 오픈을 방지하므로, 이중 결제 위험은 이중 취소보다 낮다.

### 반영 위치

- **WP4-2 산출물 추가**: "신청하기 버튼 클릭 시 즉시 비활성화 + 로딩 표시. 결제 완료/실패까지 유지"
- **WP5-1 산출물 추가**: "취소 확정 시 버튼 비활성화 + 로딩 표시. API 응답까지 유지"
- **WP5-1 엣지 케이스 추가**: "취소 API Route에서 status='confirmed' 재확인 → 이미 cancelled면 환불 없이 즉시 반환 (멱등성)"
- **Scenario 추가**: "신청하기 버튼 연속 클릭 시 결제창이 1번만 열림"

## 문제 5. 취소 시 환불이 실패하면 어떻게 되는지 정의가 없다 (환불 실패 처리 누락)

### 문제 상황

WP5-1(회원 셀프 취소)의 엣지 케이스에 "포트원 환불 API 호출이 실패하면 어떻게 되는가?"가 빠져있다.

다른 곳에는 환불 실패 처리가 모두 정의되어 있다:
- WP4-1 (정원 초과 환불 실패): "에러 로깅 + 일시적 오류 메시지" → 있음
- WP5-2 (운영자 삭제 환불 실패): deleting 상태 유지 + 재시도 로직 → 상세하게 있음
- **WP5-1 (사용자 취소 환불 실패): 정의 없음**

회원이 "취소 확정"을 눌렀는데 포트원 환불 API가 네트워크 오류로 실패한 경우, 두 가지 상황이 발생할 수 있다:
- **(A) 신청을 취소하고 환불은 실패**: 회원은 돈도 없고 신청도 없는 상태 → 최악
- **(B) 취소를 안 하고 에러 메시지**: 회원은 "다시 시도해주세요" → 안전

결과: (A)로 구현되면 회원이 돈을 잃는 상황이 발생한다.

### 해결방안: 환불 실패 시 취소하지 않고 에러 메시지 표시

환불 API 호출이 실패하면 registrations 상태를 변경하지 않는다 (confirmed 유지). 프론트에 "환불 처리 중 문제가 발생했습니다. 잠시 후 다시 시도해주세요." 안내 메시지를 표시한다. 돈이 돌아왔다는 게 확인된 후에만 신청을 취소하는 것이 원칙이다.

### 반영 위치

- **WP5-1 엣지 케이스 추가**: "포트원 환불 API 호출 실패 시 → registrations 상태 변경 없음 (confirmed 유지) + 에러 메시지 표시 + 재시도 유도"
- **기술 스택 §4-2 흐름 3**: ②~③ 사이에 환불 실패 분기 추가. "환불 API 실패 → ③번(DB 상태 업데이트) 진행하지 않음. confirmed 유지 + 에러 메시지 반환"
- **Scenario 추가**: "취소 확정 후 환불 API 실패 → 신청 상태 유지 + 에러 안내"

## 추가 권장사항 (MVP 출시 후 개선)

아래 항목은 MVP 출시에 치명적이지 않으나, 운영 중 개선하면 좋은 항목이다.

| 항목 | 설명 | 판단 |
|------|------|------|
| 내 신청 목록 중복 표시 | 취소 후 재신청 시 같은 모임이 두 줄로 보임 | 발생 빈도 낮음. 출시 후 개선 |
| 완료 화면 네비게이션 | 확정/취소 완료 후 "목록으로 돌아가기" 버튼 | PRD에 한 줄 추가 |
| 참가비 수정 충돌 | 수정 중 동시 결제 시 금액 불일치 | 모임 삭제 후 재생성으로 운영 대응 |
| 빈 상태 UI | WP4-3 산출물에 빈 상태 명시 누락 | 시나리오에는 있으므로 경미 |
