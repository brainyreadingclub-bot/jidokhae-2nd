### 수정 1-C. §4-2 흐름 2 결제 흐름 — popup/redirect 이중 경로 + Webhook 별도 분리

**위치**: 341~387행

**현재**: ①~④단계가 popup 콜백 단일 경로만 기술

**수정 원칙**: 흐름 2는 **사용자 대면 흐름**이므로, popup/redirect만 반영. Webhook은 흐름 2 아래에 별도 백업 경로로 기술

① 단계에 결제창 모드 명시:
```
① 회원이 "신청하기" 클릭
→ [프론트엔드] 포트원 SDK 호출 → PG사 결제창 표시 (popup 또는 redirect, 브라우저 환경에 따라 자동 결정)
```

② 단계에 두 경로 병기:
```
② 회원이 결제 완료
→ [popup] 포트원이 프론트엔드 콜백으로 결제 결과 전달 (payment_id 포함)
→ [redirect] 포트원이 returnUrl로 리다이렉트. 복귀 페이지에서 URL 파라미터로 payment_id 수신
```

③ 단계에 payment_id 멱등성 한 줄 추가:
```
③ 프론트엔드가 API Route 호출 ("이 결제를 검증해줘")
→ [API Route] payment_id로 기등록 여부 선행 체크 (멱등성 — 이미 등록됐으면 성공 반환)
→ [API Route] 미등록 시 포트원 REST API로 결제 검증 요청
```

④는 기존과 동일

흐름 2 아래에 Webhook 백업 경로 별도 섹션 추가:
```
**백업 경로: 포트원 Webhook**

위 ②에서 프론트엔드 콜백/redirect 복귀가 모두 실패한 경우 (인앱 브라우저 전환, 네트워크 끊김 등),
포트원이 서버 Webhook 엔드포인트(/api/webhooks/portone)에 직접 결제 완료를 통지한다.
Webhook 핸들러는 서명 검증 → payment_id 기등록 확인 → 미등록 시 결제 검증 + 등록 처리.
이미 등록된 건은 무시한다.
```

**근거**: 문제 1, 2, 3

### 수정 1-D. §4-2 흐름 2 DB Function 의사코드 — 중복 체크 추가

**위치**: 372~383행

**현재**:
```
1. SELECT ... FROM meetings WHERE id = p_meeting_id FOR UPDATE
2. SELECT COUNT(*) FROM registrations WHERE meeting_id = p_meeting_id AND status = 'confirmed'
3. IF count < capacity → INSERT / ELSE → 'full'
```

**수정**:
```
1.   SELECT ... FROM meetings WHERE id = p_meeting_id FOR UPDATE
1.5. SELECT COUNT(*) FROM registrations
     WHERE user_id = p_user_id AND meeting_id = p_meeting_id AND status = 'confirmed'
     → count > 0이면 'already_registered' 반환
2.   SELECT COUNT(*) FROM registrations WHERE meeting_id = p_meeting_id AND status = 'confirmed'
3.   IF count < capacity → INSERT / ELSE → 'full'
```

**근거**: 문제 3 (문제 상황 A)

### 수정 1-E. §4-2 흐름 3 취소+환불 — 환불 실패 분기 추가

**위치**: 389~406행

**현재**: ②(환불 호출) → ③(DB 업데이트)가 직선 흐름. 환불 실패 분기 없음

**수정**: ②~③ 사이에 분기 추가:
```
② API Route 처리
→ [API Route] 환불 금액 계산
→ [API Route] 포트원 환불 API 호출 (전액 또는 부분 환불, 0원이면 호출 생략)

②-실패. 환불 API 호출 실패 시

→ 프론트에 "환불 처리 중 문제가 발생했습니다. 잠시 후 다시 시도해주세요." 에러 메시지 반환
→ ③으로 진행하지 않음

③ 환불 성공 (또는 0원 환불인 경우)
→ (이하 기존과 동일)
```
