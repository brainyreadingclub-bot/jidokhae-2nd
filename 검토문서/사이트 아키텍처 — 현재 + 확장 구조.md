1. 사이트 아키텍처 — 현재 + 확장 구조
먼저 현재 라우트 구조와 향후 기능이 어디에 붙는지를 한눈에 보여드립니다. 녹색이 현재 MVP, 보라색이 향후 추가될 영역입니다.
  ::view-transition-group(*),
  ::view-transition-old(*),
  ::view-transition-new(*) {
    animation-duration: 0.25s;
    animation-timing-function: cubic-bezier(0.19, 1, 0.22, 1);
  }
VvisualizeVvisualize show_widget핵심 포인트는 기존 라우트 그룹 구조를 건드리지 않고 확장한다는 겁니다. (main)/ 안에 페이지만 추가하면 되고, 새로운 라우트 그룹이 필요한 건 공개 랜딩페이지((public)/) 정도뿐이에요.

2. 반응형 전략 — 모바일 코드를 건드리지 않는 PC 확장
현재 모든 페이지가 max-w-screen-sm(640px) 안에 갇혀 있습니다. PC에서 접속하면 화면 한가운데에 좁은 띠가 보이는 상태예요. 이걸 개선하는 전략은 세 단계입니다.
Phase A — 즉시 적용 가능 (코드 변경 최소): root layout의 max-w-screen-sm을 반응형으로 바꿉니다. 모바일에서는 기존 그대로, PC(md: 이상)에서는 max-w-screen-md나 max-w-screen-lg로 넓힘. BottomNav은 모바일 전용으로 유지하고, PC에서는 상단 헤더 네비게이션으로 전환. 이 단계만으로도 PC에서 답답하지 않게 됩니다.
Phase B — 레이아웃 분화: 모임 목록을 PC에서 2~3컬럼 그리드로 표시, 모임 상세를 좌우 분할(왼쪽: 모임 정보, 오른쪽: 신청/취소 액션 + 신청자 목록), 내 신청 내역을 사이드 패널로. Tailwind의 md:grid-cols-2 lg:grid-cols-3으로 모바일 코드에 영향 없이 적용.
Phase C — PC 전용 기능: 운영자 대시보드(통계, 차트), 회원 관리 화면 등 본질적으로 넓은 화면이 필요한 기능. 이건 모바일에는 없거나 축소판만 보여주는 PC 우선 페이지.

3. 기능 확장 로드맵 — 문서 기반 우선순위
PRD와 서비스 개요에 명시된 Out of Scope 항목들을 실행 가능한 순서로 정리했습니다.
  ::view-transition-group(*),
  ::view-transition-old(*),
  ::view-transition-new(*) {
    animation-duration: 0.25s;
    animation-timing-function: cubic-bezier(0.19, 1, 0.22, 1);
  }
VvisualizeVvisualize show_widget각 Wave의 설계 근거를 짚으면 이렇습니다.
Wave 1이 가장 중요합니다. 이유: MVP 출시 후 회원들이 실제로 쓰기 시작하면 "PC에서 보기 불편하다", "리마인드가 없어서 잊는다", "마감이라 신청 못 했는데 대기할 수 없다"가 가장 먼저 나올 불만입니다. 이 세 가지는 전부 운영상의 마찰을 줄이는 기능이에요. 알림톡(Solapi)은 재윤님이 이미 Kakao 템플릿 등록까지 해놓은 상태이므로 기술적 준비도 되어 있습니다.
Wave 2는 서비스 개요 3-1의 비전 후반부 — "꾸준히 함께했는지 느낄 수 있게"에 해당합니다. 배지, 활동 기록, 콩 포인트, 내 책장. 이건 회원이 서비스에 충분히 익숙해진 후에야 의미가 있어요.
Wave 3은 대외 확장입니다. 랜딩페이지(신규 회원 유입), 관리자 대시보드(운영 고도화), 운영진 권한 위임(확장 운영), 대구 확장. 이건 기존 250명이 잘 쓰고 있다는 검증이 된 후에.
Wave 4는 지독해 2nd 플랫폼이 SaaS로 진화하는 단계입니다. AI 챗봇, 챌린지 시스템, 멀티 커뮤니티. 이건 재윤님의 장기 비전(brainy-club.com = 커뮤니티 운영 SaaS)과 맞닿는 부분.

핵심 설계 원칙 요약
기존 서비스에 문제가 생기지 않는 이유: 모든 확장이 "기존 코드를 수정"하는 게 아니라 "새 파일을 추가"하는 방식이기 때문입니다. Next.js App Router의 라우트 그룹, Tailwind의 반응형 클래스, Supabase의 테이블 추가 — 전부 additive(추가적)이지 destructive(파괴적)이지 않습니다.
PC 뷰의 핵심 판단: 지금 모바일에서 잘 작동하는 것을 PC에서도 잘 작동하게 만드는 것이지, PC 전용 기능을 새로 만드는 게 아닙니다. Wave 1에서는 같은 콘텐츠를 넓은 화면에 맞게 재배치하는 것만으로 충분합니다.
