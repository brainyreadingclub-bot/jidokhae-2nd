**재평가**:

**결론**: 카피 유지. **변경 안 함**. 다음 세션에서 "입금 완료" 카피 의심이 다시 들면 이 문서를 확인할 것.

## 3. 확정된 의사결정 4건

### 3.1 ✅ 배너 관리는 editor도 권한 허용

- `src/components/admin/adminMenu.ts`의 `admin/banners`에서 `adminOnly: true` 제거.
- 정산·사이트 설정만 admin 전용으로 유지. 회원 개인정보(phone/email) 열람도 admin만.
- 한 줄(`admin/quotes`)은 원래부터 editor도 가능 — 변경 없음.

### 3.2 ✅ 회원 전화번호·이메일은 admin만 열람

- `src/app/(admin)/admin/members/page.tsx`의 서버 쿼리에서 `viewerRole === 'editor'`일 때 `phone`, `email` 컬럼을 SELECT에서 제외.
- UI(`MemberList.tsx:140-157`)는 이미 viewerRole 분기가 있으므로 서버에서 데이터만 차단하면 완성.
- React DevTools / 네트워크 탭으로도 editor는 개인정보 조회 불가.

### 3.3 ✅ 변호사 자문 불필요 — privacy 쿠키 안내 문구 + 세무서 상담으로 대체

- 변호사 자문 비용(50~100만원) 생략.
- `src/app/policy/privacy/page.tsx`에 "서비스 이용 분석을 위해 쿠키를 사용함" 한 줄 추가.
- 통신판매업 신고 면제(간이과세자), 사업자 등록은 이미 완료 — 관할 세무서 상담으로 보완 가능.

### 3.4 ✅ 입금 확인 원자성 확보 — `admin_confirm_transfer` DB Function

- 운영자가 "입금 확인" 눌렀을 때의 정원 체크 + 상태 업데이트를 **DB Function으로 원자화**.
- 기존 `register_transfer(p_user_id, p_meeting_id, p_paid_amount)` 패턴 그대로 준거.
- 새 함수 시그니처: `admin_confirm_transfer(p_registration_id UUID) RETURNS TEXT`
- 반환값: `'success' | 'not_found' | 'not_pending' | 'capacity_full'`
- 내부: FOR UPDATE로 meeting + registration 락 → 정원 재검증 → status UPDATE.
- 현재 운영자 1명일 때는 무해하지만 **editor 추가 시점부터 동시 확인이 발생할 수 있어 예방 차원 필수**.

## 6. 회원 안전 보장 7단계 (커밋 4 배포 시)

총괄책임자 지시 "**어떤 일이 있어도 문제 생기지 않도록**"을 반영한 안전장치.

1. **Vercel Preview 검증 필수**: Preview URL로 신청/취소/환불 플로우 1회 실제 시험 → 이상 없음 확인 후 prod 배포.
2. **배포 시간대**: 평일 오전 (독서모임 활동 저녁/주말 회피). 금요일 배포 금지.
3. **Preview = prod DB 주의**: 테스트 계정만 사용. 테스트 모임 즉시 삭제. 카드결제 테스트 금지(계좌이체만).
4. **각 커밋 독립 revert**: 커밋 2/3/4 분리. 이상 시 `git revert -m 1 <sha>`로 2~3분 내 복구.
5. **Forward-compatible DB**: `admin_confirm_transfer` 함수 추가는 기존 코드에 영향 없음. 코드 revert해도 함수는 남되 미사용.
6. **cron 배포 후 모니터링**: meeting-remind 병렬화 배포 다음 날 저녁 7시 실시간 로그 감시.
7. **회원 보이는 변경 최소화**: 문자열 추가 + SELECT 컬럼 제한만. 로직 변경 없음 → 오동작 이론적으로 불가능.
