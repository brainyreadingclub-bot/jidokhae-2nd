# PWA Service Worker 운영 가이드 (2026-04-28)

## 배경

2026-04-28에 PWA Service Worker(`@serwist/next` configurator mode)를 도입함. 모바일 PWA 콜드스타트 FCP 단축이 목표(P75 3.54s → 2.5s 안팎). asset(JS/CSS/font/icon)을 SW 캐시에 저장해 두 번째 방문부터 네트워크 왕복을 줄인다.

**중요한 원칙**: HTML, API, 인증 데이터는 절대 SW 캐시에 들어가지 않는다. 캐시되는 건 정적 자산(`/_next/static/*`, font CDN, PWA 아이콘)뿐. 따라서 회원·운영자 모두 데이터는 항상 fresh.

## 구성 파일

| 파일 | 역할 |
|---|---|
| `jidokhae-web/src/sw.ts` | SW 소스 (캐시 전략 정의) |
| `jidokhae-web/serwist.config.js` | Serwist build 설정 |
| `jidokhae-web/src/app/serwist.ts` | SerwistProvider re-export ('use client') |
| `jidokhae-web/src/app/layout.tsx` | SerwistProvider 마운트 (dev 모드 비활성) |

빌드 산출물: `jidokhae-web/public/sw.js` (39KB, 47개 URL precache, 1.29MB) — git ignore 됨.

## 캐시 전략 요약

| 매처 | 전략 | 의도 |
|---|---|---|
| `/api/*` | NetworkOnly | 결제·신청·환불 절대 캐시 금지 |
| `/auth/*` | NetworkOnly | PKCE 토큰 |
| `/_next/data/*` | NetworkOnly | RSC payloads |
| HTML navigation | NetworkFirst (3s timeout) | 항상 fresh, 오프라인 fallback |
| `/_next/static/*` | CacheFirst | 빌드 hash로 자동 갱신 |
| Font CDN (jsdelivr, googleapis, gstatic) | CacheFirst (1년) | 거의 안 변함 |
| PWA 아이콘 | CacheFirst | 거의 안 변함 |
| `/manifest.webmanifest` | StaleWhileRevalidate | 수정 시 갱신 가능 |
| 일반 이미지 | StaleWhileRevalidate (30일/64개) | 적당한 신선도 |

## 화면 이상 신고 받았을 때 첫 5분 진단

### 1단계 — 사용자에게 물어볼 것

- 어디서 열었나요? **홈 화면 PWA / 브라우저 탭 / 카카오톡 인앱**
- 폰 종류? **iPhone / Android**
- 언제부터 이상했나요? **방금 / 오늘 / 어제부터**
- 어떤 화면이 이상한가요? **빈 화면 / 깨진 레이아웃 / 옛날 데이터 / 버튼 안 먹음**

### 2단계 — SW 캐시 의심 시 사용자에게 안내할 알림톡 템플릿

```
[지독해] 화면 표시 문제 안내

앱을 강제 종료하고 다시 열어주세요. 그래도 이상하면 홈 화면의 [지독해] 아이콘을 길게 눌러 삭제 후, https://brainy-club.com 에서 다시 [홈 화면에 추가]를 눌러주세요.

문제가 계속되면 회신 부탁드립니다.
```

### 3단계 — 운영자 디버깅

데스크탑 Chrome으로 같은 도메인 접속 → DevTools (F12):

1. **Application 탭 → Service Workers** — 현재 SW 버전(scriptUrl) / 활성 상태(activated and is running) 확인
2. **Application → Cache Storage** — 어떤 캐시(`pages`, `next-static`, `fonts`, `app-icons`)에 무엇이 있는지 확인
3. **Network 탭** → "Disable cache" + "Bypass for network" 체크 후 재현 시도 → 문제 사라지면 SW 캐시 문제 확정

## SW Kill Switch (긴급 정지)

**언제 사용**: 캐시 버그가 광범위 신고로 번질 때. 24시간 후 모든 사용자 폰의 SW가 unregister됨.

**절차**: `jidokhae-web/src/sw.ts`를 다음 빈 SW로 임시 교체 + 배포:

```typescript
/// <reference lib="webworker" />
declare const self: ServiceWorkerGlobalScope

self.addEventListener('install', () => self.skipWaiting())
self.addEventListener('activate', (event) => {
  event.waitUntil(
    self.registration.unregister()
      .then(() => self.clients.matchAll())
      .then((clients) => clients.forEach((c) => {
        if ('navigate' in c) (c as WindowClient).navigate(c.url)
      }))
  )
})
```

배포 후 24시간 모니터링 → 잠잠해지면 정상 SW 복구 + 원인 분석 PR.

## 배포 시 주의사항

- 새 버전 prod 배포 후 **30분 모니터링** (Vercel Analytics 에러율 + 사용자 신고)
- 큰 변경(npm 의존성, layout, middleware) 동반 배포는 SW 영향이 클 수 있음
- 배포 직후 본인 PWA에서 새로고침 1회로 자동 갱신 동작 확인 (`skipWaiting + clientsClaim` 덕에 사용자도 자동)

## 캐시 만료 정책

| 캐시 이름 | 전략 | 최대 보관 |
|---|---|---|
| `pages` (HTML) | NetworkFirst | 무한 (offline fallback only) |
| `next-static` | CacheFirst | 무한 (build hash로 자동 갱신) |
| `fonts` | CacheFirst | 1년 / 16개 |
| `app-icons` | CacheFirst | 무한 |
| `app-manifest` | StaleWhileRevalidate | 무한 |
| `images` | StaleWhileRevalidate | 30일 / 64개 |
