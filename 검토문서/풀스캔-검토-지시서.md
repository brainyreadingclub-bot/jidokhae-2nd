## A. 돈이 걸린 문제 (배포 차단급)

### A-1. 웹훅 RPC 실패 시 고아 결제

**지적 에이전트**: #1 로직 감사관 (C-2), #3 데이터 수호자 (F1)

**지적 내용**: TossPayments 웹훅에서 `confirm_registration` RPC 호출이 에러를 반환하면, `rpcResult`가 `null as string`이 되어 이후 모든 조건분기(`success`, `waitlisted`, `already_registered`, `not_found`, `not_active`)에 매칭되지 않고, 최종적으로 `{ status: 'ok' }`를 반환한다. 결제는 이미 TossPayments에서 승인(DONE)됐지만, DB에 등록도 안 되고 환불도 안 된다.

**검토 대상 파일**: `src/app/api/webhooks/tosspayments/route.ts` (특히 88~137행 부근)

**검토 포인트**:
1. RPC 에러(`rpcResult.error`가 truthy) 시 실제 코드 경로를 추적하라. 어떤 response가 반환되는가?
2. 이 경로에서 `cancelPayment()` 등 환불 로직이 호출되는가?
3. redirect 경로(`/api/registrations/confirm`)에서는 같은 상황을 어떻게 처리하는지 비교하라 (`safeCancel` 로직 존재 여부)
4. 발생 조건: "프론트엔드 redirect도 실패 + 웹훅에서 RPC도 실패"라는 복합 조건이 맞는지 확인하라

---

### A-2. 사용자 취소 시 환불 성공 + DB 실패 불일치

**지적 에이전트**: #3 데이터 수호자 (F2)

**지적 내용**: 사용자가 취소를 요청하면 TossPayments 환불이 먼저 실행되고, 이후 DB에서 registration status를 `cancelled`로 업데이트한다. TossPayments 환불은 성공했으나 DB 업데이트가 실패하면, 돈은 돌려줬는데 시스템에서는 여전히 `confirmed` 상태가 유지된다.

**검토 대상 파일**: `src/lib/cancel.ts` (특히 환불 → DB 업데이트 순서)

**검토 포인트**:
1. `cancelPayment()` 호출과 DB status 업데이트의 순서를 확인하라
2. `cancelPayment()` 성공 후 DB 업데이트 실패 시 어떤 에러가 반환되는가?
3. 사용자가 재시도하면 자동 복구되는지 확인하라 (이미 환불된 건에 대해 `cancelPayment()`를 다시 호출하면?)
4. 대기자 승격(`promoteNextWaitlisted`)은 DB 업데이트 이후에 호출되는지 확인하라. DB 실패 시 승격이 누락되는가?

---

### A-3. 웹훅 서명 검증 누락

**지적 에이전트**: #1 (C-1), #2 (W로 자체 하향), #3, #6 — 4개 에이전트 공통 지적

**지적 내용**: CLAUDE.md에 "Signature verification required"로 명시되어 있으나 코드에 서명 검증 로직이 없다. `getPayment(paymentKey)` 호출로 결제 상태를 확인하고, `payment_id` 멱등성 체크로 중복 처리를 방지하고 있어 **실제 금전 피해 경로는 없다**는 것이 보안 공격자(#2)의 판단이다. 다만 스팸 POST로 TossPayments API rate limit이 소진될 수 있다.

**검토 대상 파일**: `src/app/api/webhooks/tosspayments/route.ts` (전체)

**검토 포인트**:
1. 현재 위조 방어 로직을 추적하라: 외부에서 임의의 `paymentKey + orderId`를 보내면 어떤 경로를 거치는가?
2. `getPayment(paymentKey)` 호출 — 존재하지 않는 paymentKey이면 TossPayments API가 어떤 응답을 주는가? 에러 핸들링이 되어 있는가?
3. 이미 처리된 paymentKey가 재전송되면 — 멱등성 체크(`payment_id` 기존 등록 확인)가 실제로 동작하는가?
4. 결론: 서명 검증 없이 현재 방어가 충분한지, 아니면 추가 방어가 필요한지 판단하라

## B. 법적 리스크 (조건부 통과급)

### B-1. 개인정보 수집 동의 절차 부재

**지적 에이전트**: #4 규정 준수관 (높음 1순위)

**지적 내용**: 카카오 로그인 시 이용약관/개인정보처리방침 동의 체크박스가 없다. 개인정보보호법 제15조(개인정보의 수집·이용) 위반 가능성.

**검토 대상 파일**: `src/components/auth/LoginClient.tsx`, `src/components/home/ProfileSetup.tsx`

**검토 포인트**:
1. 현재 로그인 플로우에서 약관 동의 UI가 있는지 확인하라
2. 카카오 OAuth 자체에서 동의를 받고 있는 부분이 있는지 확인하라 (카카오 개발자 설정에서 동의항목 설정 여부 — 코드에서 확인 불가하면 "운영 환경 확인 필요" 표기)
3. ProfileSetup에서 전화번호, 실명, 지역 등 추가 수집 시 별도 동의가 있는지 확인하라

3. 처리위탁 목록에 Solapi, Google Analytics가 포함되어 있는지 확인하라

## C. 운영 안정성

### C-1. Cron 순차 처리 타임아웃 위험

**지적 에이전트**: #6 인프라 감시자 (1순위)

**지적 내용**: `waitlist-refund`와 `meeting-remind` 크론이 `for...of` 루프로 순차 처리한다. 대기자/참가자가 30건을 초과하면 Vercel 10초 함수 타임아웃에 걸려 환불/알림이 누락될 수 있다.

**검토 대상 파일**: `src/app/api/cron/waitlist-refund/route.ts`, `src/app/api/cron/meeting-remind/route.ts`

**검토 포인트**:
1. 각 크론의 루프 구조를 확인하라 — `for...of` 순차인지, `Promise.all/allSettled` 병렬인지
2. 한 건당 외부 API 호출(TossPayments 환불, Solapi 알림톡)이 몇 회인지 확인하라
3. 250명 모임에서 실제 최대 건수를 추정하라 (한 모임당 대기자 최대 몇 명?)
4. 10초 타임아웃 안에 처리 가능한 최대 건수를 추정하라

---

### C-2. pending_transfer 자동 만료 없음

**지적 에이전트**: #2 보안 공격자 (W-5)

**지적 내용**: 계좌이체 모드에서 `pending_transfer` 상태로 등록하면 정원을 점유하지만, 미입금 시 자동 만료/취소 로직이 없다. 악의적 사용자가 미입금으로 정원을 독점할 수 있다.

**검토 대상 파일**: `src/app/api/registrations/transfer/route.ts`, `src/app/api/cron/` 하위 전체

**검토 포인트**:
1. `pending_transfer` 상태가 정원 계산에 포함되는지 확인하라 (`confirm_registration` RPC 또는 별도 transfer RPC에서 capacity 체크 방식)
2. 자동 만료 크론이 존재하는지 확인하라
3. 운영자가 수동으로 `pending_transfer`를 취소할 수 있는 UI/API가 있는지 확인하라

---

### C-3. 계좌이체 대기자 크론 환불 시 알림톡 미발송

**지적 에이전트**: #1 로직 감사관 (W-6)

**지적 내용**: `waitlist-refund` 크론에서 계좌이체(`payment_method = 'transfer'`)인 경우 `continue`로 건너뛰는데, 그 전에 알림톡 발송이 없다. 카드결제 대기자는 환불 + 알림톡을 받지만, 계좌이체 대기자는 아무 통보 없이 처리된다.

**검토 대상 파일**: `src/app/api/cron/waitlist-refund/route.ts`

**검토 포인트**:
1. `continue` 분기의 정확한 위치와 조건을 확인하라
2. `continue` 전에 어떤 처리가 되는지 (DB status 변경은? 알림톡은?)
3. 계좌이체 대기자의 미승격 환불은 어떤 경로로 처리되는지 (수동? 별도 크론?)

---

### C-4. Rate Limiting 부재

**지적 에이전트**: #2 보안 공격자 (W-1), #6 인프라 감시자

**지적 내용**: 결제, 취소, 웹훅 등 모든 API에 rate limiting이 없다.

**검토 대상 파일**: `src/middleware.ts`, `next.config.ts`

**검토 포인트**:
1. 미들웨어에 rate limiting 로직이 있는지 확인하라
2. Vercel 플랫폼 자체에서 제공하는 rate limiting이 있는지 확인하라 (코드에서 확인 불가하면 "운영 환경 확인 필요" 표기)
3. 250명 규모에서 실질적 위험도를 판단하라 — 내부 커뮤니티인지, 외부에 공개된 서비스인지

---

### C-5. Security Headers 미설정

**지적 에이전트**: #2 보안 공격자 (W-2), #6 인프라 감시자

**지적 내용**: `next.config.ts`에 보안 헤더가 설정되어 있지 않다. X-Frame-Options, X-Content-Type-Options, Strict-Transport-Security 등.

**검토 대상 파일**: `next.config.ts`

**검토 포인트**:
1. 현재 `next.config.ts`의 내용을 확인하라
2. Vercel이 기본 제공하는 보안 헤더가 있는지 확인하라 (코드에서 확인 불가하면 "운영 환경 확인 필요" 표기)
