# JIDOKHAE PRD (Product Requirements Document)

> 이 문서는 AI 코딩 에이전트가 구현 중 판단 기준으로 사용하는 문서입니다.
> "무엇을(What)" 만들고 "왜(Why)" 만드는지를 정의합니다. "어떻게(How)" 구현하는지는 포함하지 않습니다.
> 애매한 상황에서는 서비스 개요의 사용원칙(쉬워야/모여야/보여야)으로 돌아가 판단하세요.

## 1. 배경 및 맥락

### 현재 상황

지독해는 경주/포항에서 운영되는 250명 규모의 독서모임이다. 현재 모임 운영은 다음과 같이 이루어지고 있다.

- 일정 공지: 카카오톡 오픈채팅방에 텍스트로 게시
- 참가 신청: 오픈채팅방에 댓글로 "신청 완료"
- 결제: 입금자명을 "○요일 닉네임"으로 변경 후 계좌이체
- 입금 확인: 운영자가 수동으로 은행 앱에서 확인
- 취소/환불: 운영자에게 개인 연락 → 운영자가 입금 내역 확인 후 처리

### 문제

1. **일정이 안 보인다** — 카톡 대화에 묻혀서 모임 일정을 찾기 어렵다. 리마인드를 보낼 때마다 신청이 몰리는 건, 평소에는 정보에 접근하기 어렵다는 뜻이다.
2. **신청이 번거롭다** — 입금자명 변경 → 계좌이체 → 댓글 신청의 3단계를 거쳐야 한다.
3. **결제 확인이 수동이다** — 운영자가 매번 은행 앱에서 입금 내역을 일일이 확인해야 한다.
4. **취소/환불이 느리다** — 개인 연락 → 확인 → 처리까지 시간이 걸린다.

### 이 제품이 필요한 이유

이 불편함들이 참여의 문턱을 높이고 있다. 일정이 눈에 잘 보이고, 신청과 결제가 매끄러우면, 회원들이 더 자연스럽게 모임에 참여하게 된다.

## 2. 목적 및 목표

### 해결하려는 문제

회원의 모임 참여 과정(일정 확인 → 신청 → 결제)에서 발생하는 마찰을 제거한다.

### 목표

| 구분 | 현재 | 목표 |
|------|------|------|
| 결제 화면 진입까지 클릭 수 | 카톡 스크롤 + 입금자명 변경 + 이체 | 3클릭 이내 (일정 목록 → 모임 선택 → 신청하기) |
| 운영자 입금 확인 소요 시간 | 건당 수동 확인 | 0 (간편결제로 자동 처리) |
| 일정 확인 소요 시간 | 카톡 스크롤하여 찾기 | 웹 접속 즉시 확인 (3초 이내) |
| 취소/환불 처리 | 운영자에게 연락 → 수동 확인 → 수동 환불 | 회원이 직접 취소 → 자동 환불 |

## 3. 주요 사용자 스토리

### 회원

- "기존 회원으로서, 웹에 접속하면 다가오는 모임 일정을 한눈에 보고 싶다. 그래서 참여할 모임을 빠르게 결정할 수 있다."
- "기존 회원으로서, 마음에 드는 모임을 누르고 바로 결제까지 끝내고 싶다. 그래서 입금자명 바꾸고 이체하는 번거로움 없이 참여를 확정할 수 있다."
- "기존 회원으로서, 내가 신청한 모임 내역을 언제든 확인하고 싶다. 그래서 뭘 신청했는지 헷갈리지 않는다."
- "기존 회원으로서, 참여를 취소하고 싶을 때 환불 규정을 확인하고 바로 취소하고 싶다. 그래서 운영자에게 연락하고 기다리지 않아도 된다."

### 운영자

- "운영자로서, 모임을 생성하면 회원들이 알아서 신청하고 결제하는 상태를 원한다. 그래서 입금 확인에 쓰던 시간이 사라진다."
- "운영자로서, 신청자 목록과 결제 상태를 한 화면에서 확인하고 싶다. 그래서 구글시트와 은행앱을 왔다갔다 할 필요가 없다."
- "운영자로서, 모임 정보를 쉽게 수정하고 삭제하고 싶다. 그래서 일정 변경이 생겨도 빠르게 반영할 수 있다."

## 4. 성공 지표

### 정량적 지표

- 결제 화면 진입까지 클릭 수: 3회 이내 (우리 서비스 내, PG사 결제창 내 클릭은 제외)
- 운영자 입금 확인 건수: 월 0건 (간편결제로 자동 처리)
- 웹 접속 후 일정 확인까지 소요 시간: 3초 이내
- 취소/환불 처리 시 운영자 개입 건수: 월 0건 (자동 처리)

### 정성적 지표

- 회원 피드백: "모임 신청이 편해졌다"
- 운영자 체감: "입금 확인 안 해도 된다"
- 일정 디자인에 대한 반응: "참여하고 싶게 생겼다"

## 5. 핵심 기능 & 제외 항목

### 5-1. 반드시 포함 (MVP)

**회원 기능**

| 기능 | 설명 |
|------|------|
| 카카오 로그인 | 별도 회원가입 없이 카카오 계정으로 로그인 |
| 모임 일정 목록 | 다가오는 모임이 한눈에 보이는 화면. 참여하고 싶게 생긴 디자인. 마감 모임은 "마감" 뱃지 표시, 내가 신청한 모임은 "신청완료" 표시. **회원에게는 오늘(포함) 이후 active 모임만 표시. 운영자에게는 과거 모임 및 deleting 상태 모임도 표시 (deleted는 제외)** |
| 모임 상세 페이지 | 날짜, 시간, 장소, 남은 자리 등 모임 정보 확인. **현재 confirmed 신청 유무와 모임 시점에 따라 버튼이 달라진다** (아래 버튼 분기 참조). cancelled 레코드만 있는 경우 미신청과 동일 취급 |
| 모임 신청 + 결제 | "신청하기" 버튼을 누르면 바로 PG사 결제창으로 이동. 결제 완료 = 신청 확정. 결제 없이 신청만 존재하는 상태는 없다. **결제 실패(취소, 한도 초과, 오류 등) 시 모임 상세 페이지로 돌아가며 "결제가 완료되지 않았습니다. 다시 시도해주세요." 안내 메시지 표시** |
| 정원 마감 처리 | 정원이 찬 모임은 "마감" 뱃지 표시 + 신청하기 버튼 비활성화. 모임 상세에서도 "마감" 상태를 명확히 표시 |
| 내 신청 내역 | 내가 신청한 모임 목록과 결제/취소 상태를 확인할 수 있는 화면. **과거 모임 이력도 포함** (참여 완료 / 취소됨 상태 표시) |
| 셀프 취소 + 자동 환불 | 취소 버튼 → 환불 규정 안내 → "취소를 확정하시겠습니까?" 확인 → 확정 시 신청 취소 + PG사 환불 API로 자동 환불. 환불 금액은 환불 규정(3일 전 100% / 2일 전 50% / 이후 환불 불가)에 따라 자동 계산. **모임 날짜 다음 날부터는 취소 불가. 취소 후 자리가 남아있으면 재신청 가능** |

**운영자 기능**

| 기능 | 설명 |
|------|------|
| 모임 생성 | 모임명, 날짜, 시간, 장소, 정원, 참가비를 입력하여 모임 생성 |
| 모임 수정 | 생성된 모임 정보 수정 (단, 참가비 변경 시 기존 결제 건에는 영향 없음) |
| 모임 삭제 | 결제된 신청자가 있는 모임도 삭제 가능. 삭제 시 모든 신청자에게 100% 자동 환불 처리 후 삭제. 운영자에게 "신청자 N명에게 환불됩니다" 확인 절차 필수 |
| 신청자 목록 | 모임별 신청자 이름, 결제 상태, 취소 여부를 한 화면에서 확인 |

## 6. UX/디자인 가이드

> 구체적인 디자인 톤/시스템은 별도 문서로 분리한다. PRD에는 UX 원칙과 핵심 사용자 흐름만 정의한다.

### 6-1. UX 원칙

- **3초 규칙**: 웹 접속 후 3초 안에 다가오는 모임 일정을 파악할 수 있어야 한다
- **3클릭 규칙**: 일정 목록에서 결제 화면 진입까지 3클릭 이내. PG사 결제창 내의 클릭은 우리가 통제할 수 없으므로 제외
- **모바일 우선**: 회원 대부분이 모바일로 접속한다. 모바일 웹을 우선 설계하고, PC는 기본 대응
- **일정이 참여하고 싶게 보여야 한다**: 일정 목록은 단순 텍스트 나열이 아니라, 시각적으로 매력적인 디자인이어야 한다
- **내 상태가 명확해야 한다**: 내가 신청한 모임, 결제 상태, 취소 상태가 항상 분명하게 보여야 한다
- **마감 상태가 즉시 인지되어야 한다**: 정원이 찬 모임은 목록에서도 상세에서도 "마감"이 바로 보여야 한다. 회원이 상세 페이지까지 들어갔다가 마감인 걸 알게 되면 안 된다

### 6-2. 모임 상세 페이지 버튼 분기

모임 상세 페이지의 버튼은 **"현재 confirmed 신청이 있는가"**와 **"모임 시점"**으로만 결정한다. cancelled 레코드만 있는 경우는 미신청과 동일하게 취급한다 (재신청이 자연스럽게 가능해짐). "취소됨" 상태 표시는 내 신청 목록 화면에서만 보여준다.

| 조건 | 표시 |
|------|------|
| 모임 전 + confirmed 없음 + 자리 있음 | "신청하기" 버튼 (활성) — 첫 신청이든 재신청이든 동일 |
| 모임 전 + confirmed 없음 + 마감 | "마감" (비활성) |
| 모임 당일 이전(포함) + confirmed 있음 | "취소하기" 버튼 (활성) |
| 모임 지남 + confirmed 있음 | "참여 완료" 텍스트, 버튼 없음 |
| 모임 지남 + confirmed 없음 | 버튼 없음 (과거 취소 건 조회 시) |

> 운영자에게는 위 버튼 분기와 별도로 "수정" / "삭제" 버튼이 추가 표시된다. 신청자 목록도 운영자에게만 표시된다.

**내 신청 목록의 상태 뱃지** (모임 상세 버튼과는 별개):

| 조건 | 뱃지 |
|------|------|
| confirmed + 모임 전 | "신청완료" |
| confirmed + 모임 지남 | "참여 완료" |
| cancelled (시점 무관) | "취소됨" |

### 6-3. 핵심 사용자 흐름

**회원 — 모임 신청 흐름**

```
웹 접속 (카카오 로그인)
  ↓
모임 일정 목록 화면 (마감된 모임은 "마감" 뱃지 표시, 내가 신청한 모임은 "신청완료" 표시)
  ↓
모임 선택 (카드 클릭) ← 1클릭
  ↓
모임 상세 화면 (정보 확인 + 남은 자리 표시)
  ↓
"신청하기" 버튼 ← 2클릭
  ↓
PG사 결제창 진입 ← 3클릭 (여기까지가 우리 서비스 범위)
  ↓
결제 완료 → 신청 확정 화면 ("신청이 완료되었습니다" + 모임 정보 요약)
  ↓
모임 일정 목록으로 복귀 (해당 모임에 "신청완료" 표시)
  ↓
내 신청 내역에서 확인 가능
```

> **핵심 규칙: 결제 완료 = 신청 확정.** 결제 없이 신청만 존재하는 상태는 없다. 결제에 실패하면 신청은 확정되지 않는다.

> **결제 실패 시**: 모임 상세 페이지로 돌아가며 "결제가 완료되지 않았습니다. 다시 시도해주세요." 안내 메시지를 표시한다. 별도 에러 페이지 없이 바로 재시도 가능.

**회원 — 취소 흐름**

```
내 신청 내역에서 취소할 모임 선택
  ↓
모임 상세 화면 (신청한 상태로 표시)
  ↓
"취소하기" 버튼
  ↓
환불 규정 안내 표시
  - 3일 전: 100% 환불
  - 2일 전: 50% 환불
  - 이후: 환불 불가
  + "단무지에게 1대1톡으로 문의하세요" 안내 표시
  ↓
"취소를 확정하시겠습니까?" 최종 확인
  ↓
확정 시: 신청 상태 → 취소, PG사 환불 API 호출 (규정에 따른 금액 자동 계산)
  ↓
취소 완료 화면 (환불 예정 금액, 환불 예상 소요 시간 안내)
```

> **환불 불가 기간(모임 전날~당일)에도 취소 자체는 가능하다.** 단, 환불 금액이 0원임을 명확히 안내하고, 회원이 이를 확인한 후 취소를 확정한다.

> **모임 날짜 다음 날부터는 취소 자체가 불가하다.** 모임이 이미 종료되었으므로 취소하기 버튼을 표시하지 않는다.

> **취소 후 재신청**: 취소한 모임이라도 아직 날짜가 지나지 않았고 자리가 남아있으면 다시 신청할 수 있다. 새로운 결제가 진행된다.

**운영자 — 모임 관리 흐름**

```
운영자 화면 접속
  ↓
모임 생성 (모임명, 날짜, 시간, 장소, 정원, 참가비 입력)
  ↓
회원들이 신청/결제
  ↓
신청자 목록 + 결제 상태 + 취소 내역 확인
```

**운영자 — 모임 삭제 흐름**

```
모임 삭제 버튼
  ↓
"이 모임에 신청자가 N명 있습니다. 삭제 시 모든 신청자에게 100% 환불됩니다." 확인
  ↓
확정 시: 모든 신청자에게 자동 환불 → 모임 삭제
```

## 7. 시스템 요구사항

### 플랫폼

- 모바일 웹 우선 (반응형으로 PC 기본 대응)
- 앱은 추후 개발 (이번 MVP는 웹만)

### 인증

- 카카오 소셜 로그인 (별도 회원가입 없음)

### 결제

- 간편결제 지원 (토스페이먼츠 직접 연동, 카드 결제)
- 결제 완료 시 신청이 자동 확정되어야 함
- 결제 실패 시 신청이 확정되지 않아야 함
- 환불은 PG사 환불 API를 통해 자동 처리

### 환불 규칙

- 모임 3일 전까지: 결제 금액의 100% 환불
- 모임 2일 전까지: 결제 금액의 50% 환불
- 모임 전날 이후: 환불 불가 (0원). 단, 취소 자체는 가능
- 모임 삭제 시: 모든 신청자에게 100% 환불 (모임 일정과 무관)
- 환불 기준 시점: 취소 확정 시점 기준으로 모임 날짜까지 남은 일수로 판단 **(날짜 단위 계산. 모임이 3월 5일이면, 3월 2일 23:59까지 = 3일 전, 3월 3일 00:00부터 = 2일 전)**
- 모임 날짜 다음 날부터는 취소 자체가 불가하다 (모임이 이미 종료됨)

### 모임 참가비

- 모임별 가변 금액 (정기모임 10,000원이 기본이지만, 추후 다른 금액의 모임이 추가될 수 있으므로 고정값이 아닌 모임별 설정 방식)

### 성능

- 모임 일정 목록 로딩: 3초 이내
- 결제 프로세스: PG사 응답에 의존하되, 서비스 자체 지연은 최소화

## 8. 가정, 제약 사항 및 의존성

### 가정 (Assumptions)

- 회원 대부분은 모바일(카카오톡 내 링크 클릭)로 접속한다
- 회원 대부분은 카카오 계정을 보유하고 있다
- 동시 접속 사용자 수는 많지 않다 (250명 규모 커뮤니티)
- 모임 종류는 당분간 정기모임만 운영한다
- 취소/환불 건수는 월 5~7건 수준으로, PG사 환불 API 호출 빈도가 높지 않다

### 제약 사항 (Constraints)

- 코딩은 AI 에이전트(Claude Code)를 통해 진행한다. 문서가 명확해야 AI가 올바르게 구현한다
- MVP는 최소 범위로 빠르게 출시하는 것이 최우선이다. 완벽함보다 실행 가능성을 우선한다
- 결제 및 환불 시스템은 PG사 API에 의존한다

### 의존성 (Dependencies)

- 카카오 로그인 API
- 토스페이먼츠 결제/환불 API (직접 연동)
- 카카오톡 1대1 채팅 링크 (취소 화면에서 운영자 문의용)

## 9. 릴리즈 계획

### MVP 출시 (이번 릴리즈)

핵심 흐름이 완전히 동작하는 상태로 출시한다: **일정 확인 → 신청/결제 → 취소/환불**

출시 기준:
- 회원이 카카오 로그인으로 접속할 수 있다
- 모임 일정 목록이 보인다 (마감 상태 포함)
- 모임을 선택하고 결제까지 완료하면 신청이 확정된다
- 내 신청 내역을 확인할 수 있다
- 회원이 직접 취소할 수 있고, 환불 규정에 따라 자동 환불된다
- 운영자가 모임을 생성/수정/삭제할 수 있다
- 운영자가 신청자 목록과 결제/취소 상태를 확인할 수 있다

### 추후 릴리즈 (우선순위는 운영하며 결정)

- 대기 신청 / 자동 승격
- 알림톡 / 리마인드
- 소속감 기능 (배지, 칭찬, 활동 기록, 내 책장)
- 신규 회원 후킹 랜딩페이지
- 관리자 대시보드 (통계/인사이트)
- 운영진 권한 위임
