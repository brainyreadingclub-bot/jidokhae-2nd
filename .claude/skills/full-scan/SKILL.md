# 검증 에이전트 팀 (Verification Agent Squad)

7명의 독립 검증 에이전트가 각자의 관점에서 코드를 검증하고, 종합 판단관이 결과를 통합하여 보고서를 작성한다.

## 에이전트 구성

| # | 이름 | 관점 |
|---|------|------|
| 1 | 로직 감사관 | 코드가 의도한 대로 정확히 동작하는가? |
| 2 | 보안 공격자 | 이 서비스를 어떻게 해킹할 수 있는가? |
| 3 | 데이터 수호자 | 데이터가 손실되거나 오염되지 않는가? |
| 4 | 규정 준수관 | 법적으로 안전한가? |
| 5 | 사용자 대변인 | 실제 사용자 경험이 괜찮은가? |
| 6 | 인프라 감시자 | 프로덕션에서 안정적으로 운영 가능한가? |
| 7 | 일관성 심판관 | 코드가 전체적으로 일관적인가? |
| 8 | 종합 판단관 | 7개 보고서를 종합하여 배포 판정 근거 제공 |

## 스팟 체크 프리셋

| 프리셋 | 에이전트 | 적합한 변경 |
|--------|---------|-----------|
| payment | #2+#3+#4 | 결제/금전 로직 |
| ui | #5+#7 | UI/UX 변경 |
| auth | #2+#1+#3 | 인증/권한 로직 |
| deploy | #6+#2 | 환경변수/배포 설정 |
| refactor | #7+#1+#2 | 전체 리팩토링 |
| db | #2+#3+#1 | DB 스키마 변경 |
| api | #1+#2+#6 | API 엔드포인트 추가 |
| external | #4+#6+#5 | 외부 서비스 연동 |

## 절차

### 0단계: 모드 결정

사용자의 요청을 분석하여 검증 모드를 결정한다.

- **풀 스캔**: "풀 스캔", "전체 검증", "full scan" 또는 단순히 "/verify"만 입력한 경우
- **스팟 체크**: "스팟 체크 payment", "결제 로직 검증해줘" 등 특정 프리셋이나 에이전트 번호를 지정한 경우

모드가 불명확하면 AskUserQuestion으로 질문한다:
- "풀 스캔(전원) vs 스팟 체크(선택) 중 어떤 모드로 실행할까요?"
- 스팟 체크 선택 시 프리셋 목록을 제시한다.

### 1단계: 프로필 확인

1. 현재 작업 디렉토리(CWD)에서 `verification-squad/profiles/` 또는 루트의 `project-profile.md`를 Glob으로 찾는다.
2. 프로필이 없으면:
   - "프로젝트 프로필이 없습니다. 검증을 위해 프로필이 필요합니다."
   - `verification-squad/templates/project-profile.md` 템플릿을 Read하여 보여주고 작성을 안내한다.
   - **여기서 멈춘다.** 프로필이 준비될 때까지 이후 단계를 진행하지 않는다.
3. 프로필이 있으면 Read하여 내용을 변수에 저장한다.

요구사항 문서도 확인한다:
- 사용자가 명시적으로 제공했거나, 대화 컨텍스트에 요구사항이 있으면 사용한다.
- 없으면 "요구사항 문서 미제공"으로 진행한다.

### 2단계: 검증 에이전트 병렬 실행

**이 단계가 핵심이다.**

1. 실행할 에이전트 번호 목록을 결정한다 (풀 스캔: 1~7, 스팟 체크: 프리셋에 따라).
2. 각 에이전트의 시스템 프롬프트를 Read한다:
   - `C:\Users\ytyt8\verification-squad\agents\0N-xxx.md`
3. **한 메시지에서 Agent 도구를 동시에 호출하여 병렬 실행한다.**

각 Agent 호출의 prompt 구성:

```
[시스템 프롬프트 전문 (agents/0N-xxx.md 내용)]

단, 보고서 저장 경로의 {{REPORT_PATH}}는 실제 경로로 치환한다.

---
아래에 프로젝트 프로필이 제공된다.
---
[프로필 내용]

---
아래에 요구사항 문서가 제공된다.
---
[요구사항 내용 또는 "요구사항 문서 미제공"]

---
검증 대상 코드 디렉토리: [CWD 또는 사용자가 지정한 경로]
이 디렉토리의 코드를 Read, Glob, Grep 도구로 읽고 검증하라.
검증 완료 후 보고서를 지정된 경로에 Write 도구로 저장하라.
```

**Agent 도구 호출 시:**
- `description`: "검증 에이전트 #N [이름]"
- `prompt`: 위 구성대로 조합한 전체 프롬프트
- 풀 스캔 시 7개를 한 번에 호출 (병렬 실행)
- 스팟 체크 시 해당 에이전트만 호출

### 3단계: 종합 판단관 실행

모든 검증 에이전트가 완료되면:

1. `reports/YYYY-MM-DD/` 디렉토리에 저장된 보고서 파일을 확인한다.
2. `agents/08-final-arbiter.md`를 Read하여 종합 판단관 프롬프트를 가져온다.
3. Agent 도구로 종합 판단관을 실행한다.
   - prompt에 보고서 디렉토리 경로와 코드 디렉토리 경로를 제공한다.
   - 종합 판단관이 보고서를 Read하고 종합 보고서를 Write한다.

### 4단계: 결과 출력

종합 보고서의 **Executive Summary** 부분을 사용자에게 출력한다:
- 배포 권고 (🟢/🟡/🔴)
- Critical 이슈 수와 한 줄 요약
- 즉시 판단 필요 항목
- 전체 보고서 경로 안내

## 보고서 저장

- 보고서 디렉토리: `verification-squad/reports/YYYY-MM-DD/`
- 같은 날짜에 여러 번 실행 시: `reports/YYYY-MM-DD-2/`, `reports/YYYY-MM-DD-3/` 증분
- 보고서 디렉토리 생성은 2단계 시작 전에 Bash로 `mkdir -p`를 실행한다.

## 중요 원칙

- **컨텍스트 격리**: 각 에이전트는 별도 Agent 서브프로세스로 실행되어 서로의 결과를 볼 수 없다.
- **독립 검증**: 에이전트 간 검증 결과를 공유하지 않는다. 종합 판단관(#8)만 보고서를 수집한다.
- **최종 결정은 사람**: 종합 보고서는 판단 근거일 뿐, 최종 결정은 사용자(총괄 책임자)가 한다.
- **프로필 부족 감지**: 프로필 정보가 부족하면 에이전트가 코드에서 추론하되, 보고서에 명시한다.

## 비용 참고

- 풀 스캔은 7+1=8개 에이전트를 실행하므로 비용이 높을 수 있다.
- 비용이 우려되면 스팟 체크 프리셋을 사용하라.
- 첫 실행 후 실제 토큰 사용량을 확인하여 운용 기준을 세우라.
