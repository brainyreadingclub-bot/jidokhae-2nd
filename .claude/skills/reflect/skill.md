# 세션 회고 (Reflect)

작업 세션이 끝날 때 호출한다. "뭘 만들었냐"가 아니라 **"어떻게 작업했냐"**에 집중하여 다음 세션에 재사용할 수 있는 교훈을 추출한다.

## 절차

### 1단계: 세션 데이터 수집 + 통계

이번 대화에서 일어난 일을 역순으로 훑는다:

- `git log --oneline -20`으로 이번 세션의 커밋 히스토리 확인
- 이번 세션의 커밋 범위를 파악한 뒤 `git diff --stat HEAD~N` (N = 세션 커밋 수)으로 변경 파일 수, +/- 라인 수 집계
- 커밋이 없으면 `git diff --stat`으로 uncommitted 변경 기준 집계
- 대화 기록에서 사용자의 수정 요청, 방향 전환, 반복 작업을 추출
- 삽질한 부분(시행착오, 되돌린 작업, 사용자가 거부한 접근법)을 식별

### 2단계: 과거 교훈 적용 점검 + 3가지 축 분석

#### 과거 교훈 적용 점검

`memory/` 디렉토리에서 `feedback_*.md` 파일들을 읽고, 이번 세션과 관련 있는 교훈을 식별한다:
- 이번 세션에서 과거 교훈이 **잘 적용된 사례**가 있으면 → "잘한 것"에 포함
- 같은 실수가 **반복된 사례**가 있으면 → "삽질한 것"에 포함 + 해당 교훈의 실효성 재검토 (교훈 내용 수정 또는 더 구체적인 규칙으로 업데이트)

#### 잘 된 접근법
- 무엇을 했고, 왜 효과적이었는가?
- 다음에도 반복할 수 있는 패턴인가?
- 예시: "기존 ModalOverlay 컴포넌트를 재사용해서 새 모달을 0부터 만들지 않았다 → 코드 일관성 + 시간 절약"

#### 삽질한 부분
- 어디서 시간을 낭비했는가?
- 근본 원인은 무엇인가? (요구사항 오해? 코드 구조 미파악? 성급한 구현?)
- 예시: "로컬에서 카카오 OAuth 테스트가 안 되는 걸 모르고 dev 서버를 띄웠다 → 배포 환경 제약을 먼저 파악했어야"

#### 다음 세션에 적용할 교훈
- 위 분석에서 도출한, 다음 세션부터 바로 적용할 수 있는 규칙
- "주의하자" 수준이 아니라 "X 상황에서는 Y를 먼저 한다" 수준으로 구체적으로

### 3단계: 메모리에 기록

분석 결과 중 **다음 세션에서도 유효한 교훈**이 있으면 메모리에 저장한다.

- 프로젝트별 메모리 시스템이 있으면 (`memory/` 디렉토리) → `feedback` 타입 메모리로 저장
- CLAUDE.md에 기록할 패턴이 있으면 → 해당 섹션에 추가 (기존 내용과 중복되면 병합)
- 일회성 교훈(이번 세션에만 해당)은 기록하지 않는다

### 4단계: 출력

아래 형식으로 사용자에게 보고한다:

```
## 세션 회고

**세션 통계:** 파일 N개 변경, +X/-Y lines, 커밋 N개

### 잘한 것
- (구체적 사례 + 왜 잘됐는지)
- (과거 교훈이 적용된 사례가 있으면 "✅ 교훈 적용: ..." 형태로 포함)

### 삽질한 것
- (구체적 사례 + 근본 원인)
- (과거 교훈이 반복 위반된 사례가 있으면 "⚠️ 교훈 반복: ..." 형태로 포함)

### 다음 세션 교훈
- (실행 가능한 규칙)

### 기록한 항목
- (메모리/CLAUDE.md에 뭘 추가했는지, 또는 "기록할 항목 없음")

### 다음 세션 핸드오프
- **현재 상태:** (방금 완료한 것, 진행 중인 것)
- **다음 할 일:** (우선순위 순, 구체적으로)
- **블로커:** (사용자 액션 필요 사항, 없으면 "없음")
```

### 5단계: 다음 세션 핸드오프 메모리 작성

회고 분석 후, **다음 세션에서 바로 작업을 이어갈 수 있도록** `project` 타입 메모리에 핸드오프 문서를 작성한다.

파일명은 **`project_next_session.md`** 고정. 매 회고마다 **덮어쓰기**하여 항상 최신 상태를 유지한다.

내용:
- **현재 진행 상태:** 방금 완료한 것, 진행 중인 것
- **다음 할 일:** 우선순위 순으로 정리 (구체적으로 — "DB 마이그레이션 SQL 실행" 수준)
- **블로커/대기 사항:** 사용자 액션이 필요한 것 (수동 테스트, SQL 실행, 환경변수 설정 등)
- **주의사항:** 다음 세션에서 주의할 기술적 이슈

MEMORY.md 인덱스에도 이 파일의 포인터를 추가/갱신한다.

### 6단계: 커밋 & 푸시 확인

회고 + 메모리 기록이 모두 완료된 후:

- `git status --short`로 uncommitted 변경 확인
- `git log --oneline @{u}..HEAD 2>/dev/null`로 unpushed 커밋 확인
- 둘 중 하나라도 있으면 사용자에게 **"커밋하고 푸시할까요?"** 확인
- 변경사항 없으면 이 단계를 건너뜀

## 주의사항

- 자기 칭찬이나 변명을 하지 않는다. 사실만 기록한다.
- 사용자가 명시적으로 지적하지 않았더라도, 돌아보면 비효율적이었던 접근법을 스스로 식별한다.
- 이미 CLAUDE.md나 메모리에 있는 내용과 중복되는 교훈은 새로 기록하지 않는다.
- 교훈이 없으면 "특별한 교훈 없음 — 이번 세션은 기존 패턴대로 순조로웠다"로 끝내도 된다.
- `project_next_session.md`는 매 회고마다 반드시 갱신한다. 이것이 세션 간 연속성의 핵심이다.
