# 지시서 사전 검토 (Review Directive)

구현 지시서를 실행하기 **전에** 호출한다. 지시서 내용에 매몰되지 않고, **이 프로젝트의 목적과 기존 시스템의 맥락을 먼저 완전히 장착한 상태에서** 지시서를 검토한다.

## 왜 이 스킬이 존재하는가

지시서가 길고 복잡할수록 Claude는 "지시서 소화하기"에 주의를 빼앗겨서, 이 프로젝트가 왜 존재하는지, 기존 코드가 왜 그렇게 생겼는지를 잊어버린다. 이 스킬은 **"코어 문서를 매번 반드시 다시 읽고, 프로젝트의 전체 맥락 위에서 지시서를 판단하는 것"**을 시스템으로 강제한다.

## 사용 흐름

1. 사용자가 `/preflight` (또는 "지시서 검토", "검토해줘") 호출
2. 이번 대화에 지시서가 아직 없으면 → **"검토할 지시서를 붙여넣어주세요."** 라고 요청하고 대기
3. 사용자가 지시서를 전달하면 → 아래 절차대로 검토 시작
4. 이번 대화에 이미 지시서가 있으면 → 바로 검토 시작

## 원칙

- 이 스킬은 **검토만** 한다. 코드를 수정하거나 구현하지 않는다.
- 검토 결과에 "문제없음"도 유효한 결론이다. 억지로 문제를 만들어내면 오히려 해롭다.
- 컨텍스트 로드를 **절대 생략하지 않는다.** 이 스킬의 존재 이유가 "매번 코어 문서를 다시 읽고 맥락을 장착하는 것"이다. 효율을 위해 건너뛰면 스킬의 목적이 무너진다.

## 절차

### 0단계: 지시서 확인

- 이번 대화에 지시서가 이미 있는지 확인한다.
- **없으면:** "검토할 지시서를 붙여넣어주세요." 라고 응답하고 **여기서 멈춘다.** 사용자가 지시서를 전달할 때까지 이후 단계를 진행하지 않는다.
- **있으면:** 1단계로 진행한다.

### 1단계: 컨텍스트 전체 로드 (생략 금지)

아래 문서들을 **반드시 모두** 읽어서 프로젝트의 목적, 규칙, 현재 상태를 완전히 파악한다. 지시서가 아무리 단순해 보여도 이 단계를 건너뛰지 않는다.

1. **CLAUDE.md** (루트) — 프로젝트 전체 스펙, 기술 결정, 비즈니스 규칙, 코드 컨벤션
2. **core/JIDOKHAE-2nd - 서비스 개요.md** — 서비스 비전, 3대 사용 원칙 (쉬워야/모여야/보여야)
3. **core/JIDOKHAE-2nd - PRD.md** — 유저 스토리, UX 플로우, 버튼 로직
4. **core/JIDOKHAE-2nd - 기술 스택 & 시스템 구조.md** — 아키텍처, 데이터 플로우, DB 스키마
5. **memory/project_next_session.md** (있으면) — 현재 진행 상태, 블로커
6. **memory/MEMORY.md** — 과거 교훈, 피드백 이력
7. **memory/feedback_*.md** — 과거 교훈 상세 (전부 읽기)

### 2단계: 지시서 대상 코드 실제 읽기

지시서가 수정/참조하는 파일들을 **실제로 읽는다.** 기억이나 CLAUDE.md 요약에 의존하지 않는다.

- 지시서가 명시한 수정 대상 파일 → 전부 읽기
- 지시서가 명시하지 않았지만 연관되는 파일 → import 관계, API 호출 관계를 따라가서 읽기
- DB 스키마 변경이 있으면 → `supabase/migration.sql`의 해당 테이블 정의 읽기

**목적:** "이 코드가 지금 왜 이렇게 생겼는지"를 이해한 상태에서 지시서를 판단하기 위함. 코드를 안 읽으면 환각 리뷰가 된다.

### 3단계: 프로젝트 지뢰 체크리스트 (필수)

이 프로젝트에서 반복된 실수 패턴을 지시서에 대해 점검한다. 해당 사항이 있을 때만 보고한다:

- [ ] **DB 마이그레이션 누락:** 새 컬럼/테이블/인덱스/함수를 사용하는데 마이그레이션 SQL 언급이 없는가?
- [ ] **enum/선택지 미확인:** 지시서에 고정 목록(지역, 상태값 등)이 있는데 충분한지 사용자 확인을 거치지 않았는가?
- [ ] **import/함수명 불일치:** 지시서가 참조하는 함수명, 컴포넌트명, 파일 경로가 실제 코드와 다른가?
- [ ] **동일 파일 다중 수정:** 여러 단계에서 같은 파일을 반복 수정하는 구조인가? (통합 가능성 검토)
- [ ] **환경변수 누락:** 새 외부 서비스/API를 사용하는데 환경변수 추가 언급이 없는가?
- [ ] **코드 확정 전 SQL:** DB CHECK 제약 등이 코드보다 먼저 실행되면 불일치가 발생하는 구조인가?
- [ ] **텍스트 수정 범위:** 텍스트 변경 지시가 있을 때 부분 교체인지 전체 교체인지 모호한가?

### 4단계: 총괄책임자 검토

프로젝트의 목적, 범위, 우선순위를 **진정으로** 이해한 총괄책임자(PM/Tech Lead) 관점에서 검토한다:

- **범위 적정성:** 이 지시서가 현재 Phase/Milestone에 맞는 작업인가? Out of scope 작업이 섞여 있지 않은가?
- **스펙 정합성:** 코어 문서(서비스 개요, PRD, 기술 스택)와 충돌하는 내용이 있는가? 충돌 시 **문서명 + 섹션**을 인용.
- **기존 구현 충돌:** 2단계에서 읽은 실제 코드와 모순되는 부분이 있는가?
- **누락 사항:** 지시서가 당연히 포함했어야 하는데 빠진 것이 있는가?
- **의존성 순서:** 작업 순서가 올바른가? 선행 작업이 빠져 있지 않은가?
- **3대 원칙 부합:** 쉬워야/모여야/보여야 원칙에 어긋나는 방향이 없는가?

### 5단계: 전문가 검토 (최대 3인)

지시서의 성격에 따라 **가장 적합한 세계 최고 수준 전문가**를 선정하고, 각 관점에서 검토한다. 2명으로 충분하면 2명만 쓴다. 3명이 최대.

전문가 선정 기준:
- 지시서가 **프론트엔드 UI** 중심이면 → UX 디자이너, 프론트엔드 아키텍트, 접근성 전문가
- 지시서가 **결제/비즈니스 로직** 중심이면 → PG 결제 전문가, 동시성/데이터 정합성 전문가, 보안 전문가
- 지시서가 **DB/인프라** 중심이면 → DBA, 클라우드 아키텍트, 성능 엔지니어
- 지시서가 **외부 연동** 중심이면 → API 설계 전문가, 장애 대응 전문가, 보안 전문가
- 지시서 내용과 무관한 분야의 전문가를 넣지 마라

각 전문가는 자신의 전문 분야 관점에서:
- **리스크:** 이 지시서대로 구현하면 발생할 수 있는 문제
- **개선 제안:** 더 나은 접근법이 있는가 (단, over-engineering 경고는 하지 않음 — 지시서의 범위를 존중)
- **빠진 고려사항:** 자신의 전문 분야에서 반드시 고려했어야 하는 것

### 6단계: 영향 범위 분석

지시서를 실행했을 때 변경이 필요한 파일/시스템을 나열한다:

- **변경 파일:** 지시서가 명시한 파일 + 지시서가 언급하지 않았지만 연쇄적으로 수정이 필요한 파일
- **DB 변경:** 새 테이블, 컬럼, 인덱스, RLS 정책, DB 함수 변경 여부
- **환경변수:** 새로 필요한 환경변수
- **배포 영향:** Vercel 설정, vercel.json, 도메인 등

### 7단계: 판정 및 출력

아래 형식으로 보고한다. 모든 발견 사항에 심각도 태그를 붙인다:
- `🔴 치명` — 이대로 진행하면 장애, 데이터 손실, 결제 오류 등 심각한 문제 발생
- `🟡 주의` — 빠진 사항이나 충돌이 있어 보완 필요
- `💡 제안` — 없어도 되지만 있으면 더 나은 접근

```
## 지시서 검토 결과

**판정:** 🟢 진행 가능 / 🟡 수정 후 진행 / 🔴 재작성 필요

### 지뢰 체크
- (해당 항목만 표시, 없으면 "해당 없음")

### 총괄책임자 검토
- [🔴/🟡/💡] (구체적 사항 — 문서명:섹션 인용 포함)

### 전문가 검토

**[전문가 1: 직함]**
- [🔴/🟡/💡] (리스크 / 개선 제안 / 빠진 고려사항)

**[전문가 2: 직함]**
- [🔴/🟡/💡] (리스크 / 개선 제안 / 빠진 고려사항)

### 영향 범위
- 변경 파일: ...
- DB 변경: ...
- 환경변수: ...

### 확인 필요 사항
1. (지시서에서 애매한 부분을 구체적 질문 형태로 제시)
2. (사용자가 바로 답할 수 있는 수준의 질문)

### 수정 권고사항
1. (구체적으로 지시서의 어느 부분을 어떻게 고쳐야 하는지)
```

**확인 필요 사항**은 지시서가 애매한 부분이 있을 때만 포함한다. 명확하면 생략.
**수정 권고사항**은 🟡/🔴 판정일 때만 포함한다. 🟢이면 생략.

## 판정 기준

- **🟢 진행 가능:** 🔴 치명 항목이 0개. 사소한 💡 제안만 있을 수 있음. 아래 **구현 진행 절차**를 안내한다.
- **🟡 수정 후 진행:** 🔴은 없지만 🟡 주의 항목이 있어서 지시서 보완 필요. 수정 권고사항을 구체적으로 제시.
- **🔴 재작성 필요:** 🔴 치명 항목이 있거나, 근본적인 접근 방향이 스펙과 심각하게 충돌. 대안을 제시.

## 구현 진행 절차 (검토 통과 후)

검토가 🟢 또는 🟡(수정 완료 후)로 통과되어 구현을 진행할 때, **반드시 별도 브랜치와 워크트리에서 작업한다.** main 브랜치에서 직접 코드를 수정하지 않는다.

### 브랜치/워크트리 생성 규칙

1. **브랜치 이름**: 지시서의 성격을 반영하는 이름으로 생성
   - 예: `phase0-content-quality`, `fix/daeun-interpretation`, `feature/saju-type-label`
2. **워크트리 사용**: Agent 도구의 `isolation: "worktree"` 옵션을 활용하거나, 수동으로 브랜치를 생성하여 작업
   ```bash
   git checkout -b <branch-name>
   ```
3. **main 직접 수정 금지**: 코드 변경은 항상 별도 브랜치에서 진행하고, 검토/승인 후 main에 병합
4. **커밋 전 확인**: 현재 브랜치가 main이 아닌지 반드시 확인 (`git branch --show-current`)

### 안내 메시지

검토 통과 시 아래 형식으로 안내한다:

```
구현을 시작할까요? 진행 시 별도 브랜치를 생성하여 작업합니다.
- 브랜치명: `<제안할 브랜치명>`
- 작업 범위: <변경 파일 요약>
```

## 주의사항

- 문제가 없으면 "문제없음"이라고 하라. 있지도 않은 문제를 만들어내면 오히려 해롭다.
- 코어 문서를 **실제로 읽고** 검토하라. 기억에 의존하지 마라.
- 대상 코드를 **실제로 읽고** 현재 상태를 확인하라. CLAUDE.md 요약에 의존하지 마라.
- 지시서가 의도적으로 기존 스펙을 변경하는 경우(예: "기존 X를 Y로 바꾼다")는 충돌이 아니라 변경이다. 변경의 영향만 분석하라.
- 검토 중 발견한 사항은 **구체적으로** — "좀 더 고려 필요" 같은 모호한 표현 금지. "middleware.ts L36의 `/policy` prefix 체크에 새 경로가 포함되지 않음" 수준으로.
- "좀 더 고려 필요", "주의가 필요함", "검토 권장" 같은 표현은 **금지**. 무엇이 문제이고 어떻게 해야 하는지까지 써라.
