# UX 캘린더 뷰 전문가 분석 리포트

> **작성일:** 2026-03-24
> **분석 범위:** 홈 화면 캘린더 뷰 도입 검토 + 전체 UX/프로덕트 피드백
> **참여 전문가:** UX/서비스 디자이너, 프로덕트 전략가, UI 엔지니어
> **결론 반영:** CalendarStrip 위클리 스트립 + 리스트 하이브리드로 구현됨 (`src/components/meetings/CalendarStrip.tsx`)

---

## 1. 현재 UX 평가 (리스트 기반 모임 뷰)

### 강점

**"쉬워야 한다" 원칙 충족도: 높음**
- 홈 페이지 접속 즉시 모임 카드 리스트 표시. `date ASC` 정렬로 가장 가까운 모임이 최상단. 3초 규칙 충족.
- 카드 한 장에 날짜, 시간, 장소, 인원, 참가비가 모두 담겨 스캔 한 번에 파악 가능.

**"보여야 한다" 원칙 충족도: 높음**
- 좌측 컬러 보더로 상태 시각 구분 (신청완료/대기중/마감/모집중)
- 뱃지가 카드 우측 상단에 고정 — 스크롤 중에도 상태 인지
- 80% 정원 도달 시 인원 텍스트 `text-accent-500` 긴급성 신호

**"모여야 한다" 원칙 충족도: 높음**
- 카카오톡에 흩어진 일정이 단일 화면에 집중
- `active` + `date >= today` 필터로 관련 없는 과거 모임 자동 제외

**디자인 시스템 품질**
- "Editorial Organic" 톤이 독서모임 정체성과 잘 어울림
- `body::before` 미세 노이즈 텍스처가 디지털 느낌을 부드럽게 중화
- 카드 hover 시 `-translate-y-px` + `shadow-md` 인터랙션 피드백 적절

### 약점

**시간 맥락 부족**
- 절대 날짜만 표시 ("3월 26일 (수)"). "내일", "D-3" 같은 상대 시간 없음.
- 사용자가 머릿속으로 오늘 날짜와 비교 계산해야 함 → 인지 부하 추가.

**모임 빈도 대비 화면 밀도**
- 월 2~8개 모임이면 카드 리스트가 화면 절반도 못 채우는 경우 발생.
- 빈 공간이 "모임이 별로 없다"는 인상을 줄 수 있음.

**시간적 분포 감각 부재**
- "이번 달에 모임이 어떻게 분포되어 있는가"라는 전체 감각을 주지 못함.
- 경주(수요일)/포항(일요일) 패턴이 리스트에서는 보이지 않음.

**내 신청 페이지와의 정보 중복**
- 홈 카드와 내 신청 RegistrationCard가 거의 동일한 정보.
- 두 화면의 차별점이 뱃지 종류뿐 → "내 신청" 탭의 존재 가치 약함.

## 2. 캘린더 뷰 분석

### 결론: 풀 먼슬리 그리드 비추. "위클리 스트립 + 리스트" 하이브리드 강력 권장.

### 풀 먼슬리 그리드의 문제 (이 서비스 맥락)

| 문제 | 설명 |
|------|------|
| 밀도 대비 효율 나쁨 | 30일 중 2~8일만 모임. 모바일 7열 그리드에서 셀당 ~80px → 모임 정보 못 담음 |
| 스크롤 vs 월 전환 | 리스트는 자연스러운 세로 스크롤. 캘린더는 월 전환 버튼 필요 → 추가 인터랙션 |
| 터치 타겟 문제 | 7x5 그리드 개별 셀이 44px 미만 → 접근성 기준 위반. 다양한 연령대에 심각 |
| 감성 파괴 | PRD "참여하고 싶게 생긴 디자인" 원칙에 역행. 캘린더는 기능적이지만 감성적이지 않음 |

### 캘린더 뷰의 잠재적 장점

- 시간 분포 시각화 ("수요일/일요일에 모임 있구나")
- 개인 일정과 매핑 ("이 날은 비니까 참석 가능")
- 빈 날짜 맥락 ("다음 모임까지 얼마나 기다려야 하는지" 시각적 체감)

## 3. 캘린더 뷰 시안 3종 비교

### 시안 A: 위클리 스트립 (Weekly Strip) — 가장 강력히 권장

```
┌──────────────────────────────────┐
│  < 3월 4주차                 >   │  ← 좌우 스와이프로 주 전환
│ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ │
│ │월│ │화│ │수│ │목│ │금│ │토│ │일│ │
│ │24│ │25│ │26│ │27│ │28│ │29│ │30│ │
│ │  │ │  │ │ ●│ │  │ │  │ │  │ │ ●│ │  ← 모임 있는 날에 점 표시
│ └──┘ └──┘ └──┘ └──┘ └──┘ └──┘ └──┘ │
├──────────────────────────────────┤
│  [해당 주의 모임 카드 리스트]        │  ← 해당 주만 필터링
└──────────────────────────────────┘
```

| 평가 항목 | 결과 |
|-----------|------|
| 정보 밀도 | 카드 리스트 그대로 유지 — 풍부한 정보 + 감성 보존 |
| 시간 분포 | 7셀 스트립으로 주간 분포 한눈에 |
| 터치 접근성 | 셀당 ~47px — 44px 기준 충족 |
| 구현 복잡도 | 낮음 (날짜 필터링만 추가) |
| "오늘" 맥락 | 하이라이트로 시간 맥락 자동 제공 |
| 고려사항 | 스와이프 시 리스트 재렌더링 → Client Component화 필요 |
| 빈 주 처리 | "이번 주는 모임이 없어요" 안내 필요 |

## 4. 기타 UX 개선 기회

### 4-1. 홈 화면

| 개선 | 설명 | 복잡도 | 영향 |
|------|------|--------|------|
| 상대 시간 뱃지 | "내일", "모레", "D-3" 추가 | 낮음 | 높음 |
| 서브 카피 | "이번 달 N개의 모임이 예정되어 있어요" | 낮음 | 중간 |
| 빈 상태 개선 | 따뜻한 메시지 + 일러스트 | 낮음 | 중간 |

### 4-2. 모임 상세

| 개선 | 설명 |
|------|------|
| confirm 페이지에 환불 규정 요약 | 취소 흐름 당혹감 사전 감소 |
| 백 네비게이션 재검토 | BottomNav가 이미 복귀 역할. 상단 백 링크 필요성 재검토 |

### 4-3. 내 신청 페이지

| 개선 | 설명 |
|------|------|

### 4-4. 대기 신청

| 개선 | 설명 |
|------|------|
| 대기 순번 표시 | "현재 대기 N번째" — 불안감 감소 |

### 4-5. 결제 실패

| 개선 | 설명 |
|------|------|
| "다시 결제하기" 버튼 | payment-fail에서 바로 재시도 (한 단계 줄임) |

## 5. 프로덕트 전략 피드백

### PMF 평가: 7/10

- "모임 예약 시스템"으로는 견고. 결제-신청-취소-환불 핵심 루프 견고.
- 그러나 "커뮤니티 홈"은 아직 아님. 모임 신청/취소할 때만 접속하는 도구 수준.
- 핵심 질문: **"모임이 없을 때도 회원이 이 서비스를 열게 만들 수 있는가?"**

### Phase 2-3 작업 우선순위 재평가

**과대 투자로 보이는 것:**
- **venues + 정산 시스템 (작업 5)** — 공간 2곳, 정산 유형 둘 다 percentage 80%. 이 규모에서 settlement_type 3종류 + venue_settlements 이력 + 정산 확정 워크플로우는 과설계. 모임 폼 연결까지만 하고 정산은 Phase 3으로.

**과소 투자로 보이는 것:**
- **회원 목록 강화 (작업 7)** — 검색+필터만으로는 부족. 실제 필요한 것:
  - 특정 회원 참여 이력 (최근 6개월 몇 번 참여)
  - 연락 안 되는 회원 파악 (전화번호 없는 회원 + 최근 미참여)
  - 모임별 참석자 엑셀 다운로드 (정산/보험/보고용)

**권장 재배치 순서:**

```
1. 작업 1+2 (상수 통합) — 기술 부채 정리 먼저
2. 작업 3 (결제/환불 표시) — 운영자 일상 최대 변화
3. 작업 8 (기간 필터) — 과거 모임 확인 필수
4. 작업 4 (site_settings) — 범위 축소 (사업자정보만 DB화)
5. 작업 7 (회원 강화) — venues보다 앞으로
6. 작업 6 (대시보드) — 정산 섹션 빼고 핵심만
7. 작업 5 (venues) — 모임 폼 연결까지만. 정산은 Phase 3
```

### 캘린더 뷰 — 프로덕트 관점

- **지금은 만들지 말 것.** 월 2~8개 모임이면 리스트로 충분.
- 빈 칸 많으면 "활동 적은 커뮤니티"로 오인될 위험.
- **대안:** "내 캘린더에 추가" 버튼(.ics 다운로드, 구글/네이버 캘린더 연동)이 ROI 훨씬 높음.
- 캘린더 뷰가 필요해지는 시점: 모임 유형 3개 이상, 주 4회 이상, 지역 확장 시.

### 빠져있는 제품 기회 (우선순위순)

| # | 기회 | 설명 | 구현 복잡도 |
|---|------|------|-----------|
| 1 | **모임 후 한 줄 감상** | 모임 끝난 후 서비스 재방문 유일한 트리거. 참석자만 작성 가능 | 낮음 |
| 2 | **참여 이력 통계** | "올해 12회 참여, 월평균 2회" — 내 신청 상단에 | 낮음 |
| 3 | **"내 캘린더에 추가"** | .ics 다운로드 또는 구글/네이버 캘린더 링크 | 낮음 |
| 4 | **공지/소식 피드** | 운영 공지를 카톡에서만 → 웹으로. 카톡 의존도 감소 | 중간 |
| 5 | **소셜 공유 카드** | 모임별 동적 OG 이미지 (날짜, 장소, 남은 자리) | 중간 |
| 6 | **신청 시 참여 동기 한마디** | 선택적 텍스트 필드. 운영자 인사이트 + 다른 회원 동기부여 | 낮음 |

### 성장/리텐션 레버

| 레버 | 설명 | 시점 |
|------|------|------|
| 참여 이력 시각화 | 연속 참여 뱃지, 연간 히트맵 | Phase 2-3에 통계 카드만이라도 |
| 한 줄 감상 | 모임 후 재방문 트리거 | 빠를수록 좋음 |
| 자동 신청 (구독) | 매주 같은 모임 자동 결제 | Phase 3 이후 (결제 복잡도) |
