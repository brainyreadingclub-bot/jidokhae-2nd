## 1. 웹 기본

### 1.1 클라이언트 / 서버

- **비유**: 클라이언트는 **손님**, 서버는 **식당 주방**. 손님이 주문하면 주방이 만들어서 준다.
- **정의**: 클라이언트 = 사용자의 브라우저(또는 앱). 서버 = 요청을 받아 처리하는 컴퓨터.
- **우리 프로젝트**: 회원이 카드를 꺼내 결제 버튼 누르는 화면 = 클라이언트. `/api/registrations/confirm`에서 결제 확인 처리 = 서버.

### 1.2 API

- **비유**: 주방에 넣는 **주문서 양식**. "어떤 형식으로 요청하고 어떤 형식으로 받는지"의 규격.
- **정의**: Application Programming Interface — 서버가 외부에 제공하는 요청 창구.
- **우리 프로젝트**: `src/app/api/` 폴더 안의 모든 `route.ts` 파일이 각각 하나의 API. 예: [registrations/confirm/route.ts](../../../jidokhae-web/src/app/api/registrations/confirm/route.ts) = 결제 확인 API.

### 1.3 라우트 (Route)

- **비유**: URL 경로. 식당의 **메뉴 번호**.
- **정의**: `https://jidokhae.com/meetings/abc123` 같은 주소 하나하나가 라우트.
- **우리 프로젝트**: `src/app/(main)/meetings/[id]/page.tsx` → `/meetings/abc123` URL로 접근. `[id]`는 동적 부분.

### 1.4 컴포넌트 (Component)

- **비유**: **조립 블록**. 작은 블록(버튼, 카드)을 모아 큰 화면을 만든다.
- **정의**: 재사용 가능한 UI 조각. React에서는 함수 하나가 블록 하나.
- **우리 프로젝트**: [MeetingCard.tsx](../../../jidokhae-web/src/components/meetings/MeetingCard.tsx)는 모임 카드 한 장을 그리는 블록. 목록 페이지는 이 블록을 여러 번 반복.

### 1.5 Server Component vs Client Component

- **비유**: **인쇄물 vs 인터랙티브 키오스크**. 인쇄물은 서버에서 미리 만들어져 와서 빠르고, 키오스크는 사용자가 만지면 반응한다.
- **정의**: Server Component는 서버에서 HTML을 완성해 보낸다. Client Component는 브라우저에서 JavaScript로 동작한다. 파일 맨 위에 `'use client'`가 붙어 있으면 Client.
- **우리 프로젝트**: [MeetingCard.tsx:1](../../../jidokhae-web/src/components/meetings/MeetingCard.tsx)은 `'use client'` (클릭 반응 필요). 대부분의 `page.tsx`는 Server (DB 조회 후 HTML 생성).

### 1.6 쿠키 (Cookie)

- **비유**: 식당에서 주는 **번호표**. 다음에 다시 와도 "내가 누군지" 기억하게 해준다.
- **정의**: 브라우저에 저장되는 작은 텍스트. 요청할 때마다 자동으로 서버에 같이 보내진다.
- **우리 프로젝트**: 카카오 로그인 후 Supabase가 세션 쿠키를 설정. [middleware.ts](../../../jidokhae-web/src/middleware.ts)는 매 요청마다 이 쿠키를 읽어 로그인 상태를 확인.

### 1.7 세션 (Session)

- **비유**: 로그인한 상태에서 페이지를 **돌아다니는 동안의 티켓**. 로그아웃하면 티켓이 회수된다.
- **정의**: 서버가 "이 사용자 지금 로그인되어 있다"고 기억하는 상태. 쿠키와 짝을 이룬다.
- **우리 프로젝트**: [middleware.ts:43](../../../jidokhae-web/src/middleware.ts)의 `supabase.auth.getUser()` 호출로 매 요청마다 세션을 갱신.

### 1.8 미들웨어 (Middleware)

- **비유**: 식당 입구의 **웨이터**. 모든 손님이 먼저 거친다. 테이블 확인, 신발 체크 등.
- **정의**: 요청이 페이지에 도달하기 전에 공통 처리를 하는 층.
- **우리 프로젝트**: [middleware.ts](../../../jidokhae-web/src/middleware.ts)가 전담. 로그인 안 한 사용자를 `/auth/login`으로 자동 리다이렉트.

### 1.9 환경변수 (Environment Variable)

- **비유**: **금고 비밀번호**. 코드에 직접 적으면 공개되니, 금고에 넣어두고 꺼내 쓴다.
- **정의**: 배포 환경마다 다르게 설정되는 값. API 키, DB 주소 등 민감한 정보.
- **우리 프로젝트**: Vercel Dashboard에 설정된 `NEXT_PUBLIC_SUPABASE_URL`, `TOSSPAYMENTS_SECRET_KEY` 등. 코드는 `process.env.X`로 접근.

### 1.10 리다이렉트 (Redirect)

- **비유**: **"여긴 문 닫았어요, 옆 가게로 가세요"** 안내문.
- **정의**: 서버가 "이 주소 말고 다른 주소로 다시 요청해"라고 응답하는 것.
- **우리 프로젝트**: 결제 완료 후 TossPayments → `/meetings/[id]/payment-redirect`로 리다이렉트. 로그인 안 된 사용자가 `/meetings`에 접속 → `/auth/login`으로 리다이렉트.

## 2. 데이터베이스

### 2.1 테이블 (Table)

- **비유**: **엑셀 시트**. 하나의 시트에 하나의 종류(회원, 모임, 신청) 데이터가 저장된다.
- **정의**: DB 안의 2차원 표. 행 = 데이터 1건, 열 = 속성.
- **우리 프로젝트**: 주요 9개: `profiles`, `meetings`, `registrations`, `notifications`, `site_settings`, `venues`, `venue_settlements`, `banners`, `book_quotes`.

### 2.2 스키마 (Schema)

- **비유**: **엑셀 시트의 양식**. 어떤 열이 있고 각 열이 어떤 타입인지 정한 설계도.
- **정의**: 테이블 구조 정의. 열 이름, 데이터 타입, 제약 조건.
- **우리 프로젝트**: [supabase/migration.sql](../../../jidokhae-web/supabase/migration.sql)이 전체 스키마의 소스. 예: `meetings` 테이블의 `capacity INTEGER NOT NULL` 같은 정의.

### 2.3 컬럼 / 컬럼 타입

- **비유**: 엑셀의 **열**과 그 열이 숫자만 받을지 글자도 받을지 정한 것.
- **정의**: 테이블의 열. `UUID`, `TEXT`, `INTEGER`, `TIMESTAMPTZ`, `BOOLEAN` 등 타입이 있다.
- **우리 프로젝트**: `meetings.fee` = `INTEGER` (정수 원화), `meetings.date` = `DATE`, `registrations.paid_amount` = `INTEGER`.

### 2.4 인덱스 (Index)

- **비유**: 책의 **찾아보기**. 페이지 다 뒤질 필요 없이 바로 해당 위치로 간다.
- **정의**: 특정 컬럼 조합의 조회 속도를 높이는 별도 자료구조.
- **우리 프로젝트**: `idx_registrations_meeting_status` — "모임 id + status" 조합 조회 빠르게. 예약이 많아져도 느려지지 않는 핵심.

### 2.5 UNIQUE INDEX / 파셜 인덱스

- **비유**: 찾아보기 중에서도 **"같은 이름 2번 등장 금지"** 규칙이 걸린 것.
- **정의**: 특정 컬럼 조합이 중복되지 않도록 강제하는 인덱스. `WHERE`로 조건을 걸면 "파셜(부분)" 인덱스.
- **우리 프로젝트**: `idx_notifications_remind_unique` — 같은 모임·사용자에 대한 리마인드 알림톡이 2번 기록되지 않게 DB 레벨에서 차단.

### 2.6 트리거 (Trigger)

- **비유**: **자동 센서**. 문이 열리면 자동으로 불 켜지는 것.
- **정의**: 특정 DB 이벤트(INSERT, UPDATE 등)가 일어나면 자동 실행되는 함수.
- **우리 프로젝트**: `on_auth_user_created` 트리거 — 카카오 로그인으로 새 유저가 생기면 `profiles` 테이블에 자동으로 프로필 행 생성. 코드가 명시적으로 하지 않아도 DB가 알아서 한다.

### 2.7 RLS (Row Level Security)

- **비유**: **사물함 지문인식**. 자기 사물함만 자기가 열 수 있게 DB가 직접 체크.
- **정의**: 테이블의 행 단위로 "누가 읽을 수 있나 / 쓸 수 있나"를 DB가 강제하는 정책.
- **우리 프로젝트**: `registrations_select_own` 정책 — 로그인한 본인의 신청 내역만 조회 가능. 관리자는 `is_admin()`으로 전체 조회. 앱 코드에서 실수로 보여줘도 DB가 막아준다.

### 2.8 RPC / DB 함수

- **비유**: DB 안에 심어둔 **자판기 버튼**. 복잡한 처리를 서버가 아닌 DB가 직접 한다.
- **정의**: Remote Procedure Call. 앱에서 함수명만 호출하면 DB가 여러 SQL을 묶어 실행.
- **우리 프로젝트**: `confirm_registration()` — 정원 체크 + 중복 체크 + INSERT를 한 번에 원자적으로 처리. [migration.sql:160](../../../jidokhae-web/supabase/migration.sql)에 정의.

### 2.9 FOR UPDATE / 행 락 (Row Lock)

- **비유**: **화장실 잠금**. 먼저 들어간 사람이 다 쓸 때까지 다른 사람은 대기.
- **정의**: 트랜잭션 안에서 해당 행을 잠가서, 동시 요청이 한 번에 하나씩만 처리되게 함.
- **우리 프로젝트**: `confirm_registration()` 내부의 `SELECT ... FOR UPDATE` (migration.sql:180 근처) — 같은 모임에 동시에 결제 요청이 5건 와도 정원이 정확히 체크된다.

### 2.10 마이그레이션 (Migration)

- **비유**: **리모델링 도면**. 기존 집 구조를 바꾸는 공사 지시서.
- **정의**: 기존 DB 스키마를 변경하는 SQL 스크립트. 신규 컬럼 추가, 인덱스 추가 등.
- **우리 프로젝트**: `supabase/migration-phase3-m7.sql` — Phase 3에서 `meetings` 테이블에 5개 컬럼 추가. Supabase SQL Editor에서 수동 실행.

## 3. 인증과 결제

### 3.1 OAuth

- **비유**: **대리 출입증**. 카카오가 "이 사람이 우리 회원 맞아요"라고 보증해주면, 우리는 카카오를 믿고 들여보낸다.
- **정의**: 제3자(카카오) 인증을 통해 우리 서비스 로그인을 처리하는 표준.
- **우리 프로젝트**: Supabase Auth ↔ 카카오 OAuth. 사용자는 비밀번호를 우리한테 주지 않는다. 카카오에 로그인하면 끝.

### 3.2 PKCE

- **비유**: OAuth 과정에 들어가는 **일회용 열쇠쌍**. 도중에 열쇠가 탈취돼도 짝이 안 맞으면 무용지물.
- **정의**: OAuth 콜백 과정의 보안 강화 기법. 브라우저가 만든 열쇠와 서버가 받은 열쇠가 일치해야 로그인 확정.
- **우리 프로젝트**: [middleware.ts:31](../../../jidokhae-web/src/middleware.ts)이 `/auth/callback` 경로만은 세션 갱신을 건너뛰는 이유가 PKCE 쿠키 보호. 건드리면 로그인이 깨진다.

### 3.3 payment_id (paymentKey)

- **비유**: **영수증 번호**. TossPayments가 발급하는 이 결제 건의 고유 ID.
- **정의**: TossPayments가 각 결제마다 부여하는 고유 식별자.
- **우리 프로젝트**: `registrations.payment_id`에 저장. 동일 `payment_id`로 재호출하면 중복 처리 않고 기존 결과 반환(멱등성).

### 3.4 orderId

- **비유**: 우리가 만든 **주문번호**. 영수증 번호(payment_id)와 달리 우리가 직접 생성.
- **정의**: 결제 요청 시 우리가 TossPayments에 넘기는 주문 식별자.
- **우리 프로젝트**: 형식 = `jdkh-{meetingId8}-{userId8}-{timestamp}`. 웹훅이 주문번호만 보고도 어느 모임·사용자인지 역추적 가능.

### 3.5 Idempotency (멱등성)

- **비유**: **엘리베이터 버튼을 10번 눌러도 한 번만 움직임.** 같은 요청을 여러 번 보내도 결과가 한 번 처리된 것과 같다.
- **정의**: 동일 요청을 반복해도 부작용이 한 번만 발생하는 속성.
- **우리 프로젝트**: [payment.ts:25](../../../jidokhae-web/src/lib/payment.ts)에서 `payment_id`가 이미 있으면 신청을 중복 처리하지 않고 기존 결과 반환. 사용자가 새로고침해도 이중 결제 안 됨.

### 3.6 웹훅 (Webhook)

- **비유**: **경비실이 보내는 SMS**. 무슨 일이 생기면 저쪽이 우리한테 알려준다.
- **정의**: 외부 서비스가 우리 서버에 "이런 이벤트 일어났어요"라고 보내는 HTTP 요청.
- **우리 프로젝트**: `/api/webhooks/tosspayments` — TossPayments가 결제 성공을 알려준다. 사용자 브라우저가 리다이렉트 도중 끊겨도 웹훅이 백업으로 처리.

### 3.7 Redirect-only 결제

- **비유**: **마트 결제는 반드시 계산대 가서** 한다. 제자리에서 카드만 긁는 방식 아님.
- **정의**: 카드 결제 시 TossPayments 페이지로 이동한 뒤 완료되면 우리 페이지로 돌아오는 방식. 팝업이나 인라인 결제 안 씀.
- **우리 프로젝트**: TossPayments SDK의 `requestPayment('카드', ...)`가 자동으로 TossPayments 페이지로 이동 → 완료 후 `/meetings/[id]/payment-redirect`로 돌아옴.

### 3.8 환불 규정 (Refund Policy)

- **비유**: 공연 티켓의 **환불 기준표**.
- **정의**: 취소 시점에 따라 환불 비율이 달라지는 규칙.
- **우리 프로젝트**: [refund.ts:12](../../../jidokhae-web/src/lib/refund.ts)의 `REFUND_RULES` — 3일 전까지 100%, 2일 전 50%, 그 이후 0%. 모임 삭제는 항상 100%.

### 3.9 pending_transfer (계좌이체 브릿지)

- **비유**: **카드 심사 통과 전 임시 결제 방식**. "돈은 나중에, 일단 자리는 맡아두기".
- **정의**: 운영 정책 상태. 계좌이체로 신청하면 "정원은 잡지만 입금은 미확정" 상태에서 대기.
- **우리 프로젝트**: `site_settings.payment_mode = 'transfer_only'`일 때만 활성화. 운영자가 입금 확인 후 `confirmed`로 전환. 카드 심사 끝나면 비활성화.

### 3.10 service_role key

- **비유**: **만능 마스터 키**. RLS 같은 잠금을 다 풀고 DB에 접근할 수 있다.
- **정의**: Supabase의 관리자 키. 환경변수 `SUPABASE_SERVICE_ROLE_KEY`로 보관.
- **우리 프로젝트**: [admin.ts](../../../jidokhae-web/src/lib/supabase/admin.ts)의 `createServiceClient()`가 사용. 서버 API 라우트에서만 쓰고, 브라우저로 노출되면 절대 안 됨.

## 4. 운영

### 4.1 Vercel

- **비유**: **Next.js 전용 임대 건물**. 코드 올리면 알아서 서빙해주고, 주소 연결해주고, 새벽에 자동 알림도 보내준다.
- **정의**: Next.js를 만든 회사의 호스팅 플랫폼.
- **우리 프로젝트**: `git push`하면 자동 배포. 환경변수, Cron, Preview 환경을 Dashboard에서 관리.

### 4.2 Preview 환경 vs prod

- **비유**: **리허설 무대 vs 본 공연장**. Preview는 PR마다 자동으로 생기는 테스트 URL.
- **정의**: prod = 실제 운영 사이트. Preview = 배포 전 미리보기 URL.
- **우리 프로젝트**: Preview도 **prod Supabase에 연결**(별도 staging 없음). 테스트 모임 만들면 실제 DB에 들어가니 조심.

### 4.3 Cron (Vercel Cron)

- **비유**: **알람 시계**. 매일 정해진 시각에 자동 실행.
- **정의**: 정해진 스케줄에 특정 API를 자동 호출해주는 기능.
- **우리 프로젝트**: [vercel.json](../../../jidokhae-web/vercel.json) — 매일 KST 18:30 대기 환불 크론, KST 19:00 리마인드 크론.

### 4.4 env var (환경변수)

- **비유**: 건물마다 **비밀번호가 다른** 설정. 본 공연장과 리허설 무대의 키가 다를 수 있다.
- **정의**: 배포 환경별로 다른 값을 주입하는 방식. 코드에 직접 쓰지 않는다.
- **우리 프로젝트**: Vercel Dashboard의 Environment Variables. Production / Preview / Development 3개 탭.

### 4.5 알림톡 / Solapi / Template ID

- **비유**: **카카오 우편 배달**. 카카오가 직접 배달 안 하므로 대행사(Solapi)를 통해 보낸다.
- **정의**: 알림톡 = KakaoTalk으로 보내는 정보성 메시지. Solapi = 카카오에 허가받은 발송 대행 업체. Template ID = 사전 심사된 메시지 양식의 ID.
- **우리 프로젝트**: 5종 사용 — 신청 확인, 리마인드, 대기 확인, 승격 확정, 미승격 환불. 각각 별도 Template ID가 env var에 저장.

### 4.6 마이그레이션 실행

- **비유**: **리모델링 공사 실행**. 도면(SQL)을 실제로 건물(DB)에 적용.
- **정의**: `supabase/migration-*.sql` 파일의 내용을 Supabase SQL Editor에서 직접 실행.
- **우리 프로젝트**: CLI 자동화 안 씀. 사람이 직접 복사·붙여넣기. prod와 Preview가 같은 DB라 한 번만 실행.

### 4.7 배포 Split (step1 / step2 / step3)

- **비유**: **건물 리모델링을 구역별로 나눠 여는 것**. 한꺼번에 다 열면 문제 생겼을 때 어디가 문제인지 모른다.
- **정의**: Phase 3 M7부터 채택한 배포 전략. 한 마일스톤을 ①안정성/백엔드 ②운영자 UI ③회원 UI 3단계로 나눠 순차 머지.
- **우리 프로젝트**: M7 = step1(stable) → step2(admin overhaul) → step3(member home). 각 step 머지 후 1~2일 모니터링.

### 4.8 Cherry-pick

- **비유**: **과자 상자에서 원하는 것만 골라 담기**.
- **정의**: 다른 브랜치의 특정 커밋만 골라서 현재 브랜치에 가져오는 Git 명령.
- **우리 프로젝트**: 배포 split에서 필수. `backup/phase3-m7-full` 브랜치에서 step1 커밋만 골라 `feat/phase3-m7-step1-stable` 브랜치로 cherry-pick.

### 4.9 Git Worktree

- **비유**: **같은 집의 복사본을 다른 주소에 열기**. 한쪽에서 공사해도 원본이 영향 안 받음.
- **정의**: 같은 레포를 다른 폴더에서 다른 브랜치로 동시 작업할 수 있게 하는 Git 기능.
- **우리 프로젝트**: Claude가 `isolation: worktree` 옵션으로 안전하게 실험. 메인 작업 방해 안 함.

### 4.10 prelaunch

- **비유**: **공연 전 전체 리허설**. lint → 타입 검사 → 테스트 → 빌드 한 번에.
- **정의**: `npm run prelaunch` 명령. 4단계 검증을 순차 실행.
- **우리 프로젝트**: [package.json:13](../../../jidokhae-web/package.json) — WP 하나 완료할 때마다 실행. 하나라도 실패하면 커밋 금지.

## 5. 개발 품질

### 5.1 TypeScript

- **비유**: JavaScript + **포장 라벨**. "이 상자엔 사과 5개 들어있어요"라고 미리 표시하면, 다른 코드가 잘못 쓸 때 경고가 뜬다.
- **정의**: JavaScript에 정적 타입을 더한 언어. 컴파일 시점에 오류를 잡아준다.
- **우리 프로젝트**: 모든 `.ts`/`.tsx` 파일. `npx tsc --noEmit` 으로 타입만 검사.

### 5.2 타입 (Type)

- **비유**: **붕어빵 틀**. 이 틀에 맞는 것만 들어간다.
- **정의**: 데이터의 모양 정의. `type Meeting = { id: string; date: string; ... }` 처럼.
- **우리 프로젝트**: `src/types/meeting.ts`, `src/types/registration.ts`. Supabase 자동 생성 안 쓰고 수동 정의.

### 5.3 Lint (ESLint)

- **비유**: **글짓기 맞춤법 검사**. 실행되지는 않지만 스타일·안티패턴을 잡아준다.
- **정의**: 코드 스타일과 잠재적 버그를 정적 분석으로 찾는 도구.
- **우리 프로젝트**: `npm run lint`. `eslint-config-next`의 core-web-vitals + typescript 규칙.

### 5.4 Build

- **비유**: **책 원고를 인쇄 파일로 변환**. 최종 납품용으로 압축·최적화.
- **정의**: 개발용 소스 코드를 배포용으로 컴파일·번들링하는 과정.
- **우리 프로젝트**: `npm run build` — Next.js가 서버/클라이언트 코드 분리, 이미지 최적화, 정적 페이지 생성.

### 5.5 Git Branch

- **비유**: **작업용 복사본**. 원본(main) 안 건드리고 옆에서 실험.
- **정의**: 같은 코드베이스의 평행 작업 공간. 완성되면 main에 병합.
- **우리 프로젝트**: `feat/phase3-m7-step2-5-hotfix` 같은 기능 단위 브랜치. `main`이 prod 소스.

### 5.6 커밋 (Commit)

- **비유**: **일기 한 줄 기록**. "이 시점에 뭘 바꿨는지" 스냅샷.
- **정의**: 코드 변경 사항 + 설명을 Git 히스토리에 기록하는 단위.
- **우리 프로젝트**: WP 하나 완료 = 커밋 하나 원칙. 메시지 형식: `feat(M7 Step 2): ...`.

### 5.7 풀 리퀘스트 (PR)

- **비유**: **승인 요청서**. "이 작업을 main에 넣어도 될까요?"를 문서화.
- **정의**: 브랜치를 main에 병합하기 전에 리뷰받는 단계.
- **우리 프로젝트**: `gh pr create`로 생성. 각 배포 step이 별도 PR.

### 5.8 /full-scan

- **비유**: **7명 전문가가 동시에 품질 감리**.
- **정의**: 이 프로젝트 전용 skill. 7개 검증 에이전트(보안, 성능, UX, DB, 타입 등)가 병렬로 코드 스캔 후 종합 리포트 생성.
- **우리 프로젝트**: WP 완료 시, 또는 사용자가 `/full-scan` 호출 시 실행. 결과는 지적의 **우선순위**와 **근거**가 함께 나옴.

### 5.9 테스트 (Vitest)

- **비유**: **부품별 품질 검사 자동화**. 변경할 때마다 자동으로 돌려서 "망가진 부품 없나" 확인.
- **정의**: 코드 조각이 의도대로 동작하는지 자동 검증하는 함수 모음.
- **우리 프로젝트**: `src/lib/__tests__/kst.test.ts`, `refund.test.ts`만 존재. E2E 테스트는 없음(수동 체크리스트로 대체).

### 5.10 프론트엔드 vs 백엔드

- **비유**: **매장 인테리어 vs 주방 설비**.
- **정의**: 프론트엔드 = 사용자 눈에 보이는 부분. 백엔드 = 서버·DB 등 뒤쪽 처리.
- **우리 프로젝트**: Next.js 안에서 둘 다 섞여 있다. `src/components/`, `src/app/.../page.tsx` = 프론트. `src/app/api/`, `src/lib/`, `supabase/` = 백엔드.
