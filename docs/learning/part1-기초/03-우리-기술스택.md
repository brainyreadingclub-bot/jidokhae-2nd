# 03. 우리 기술스택 — 왜 이걸 골랐나

> **이 챕터에서 배우는 것**
> 1. JIDOKHAE를 구성하는 6개 도구 — 뭐하는 건가
> 2. 왜 **이 도구**를 골랐고, 다른 선택지는 뭐였나
> 3. 이걸 만질 때 건드리는 파일은 어디인가

---

## 왜 이 챕터가 중요한가

도구 선택은 "유행" 문제가 아니다. **프로젝트의 제약**(운영자 1인, 서울 외 지역 250명, 카드 결제 심사 대기)을 반영한 결정이다.

여기서 배경을 알면:
- 에이전트가 **"X를 Y로 바꾸자"**고 제안할 때 "이 제약 고려했어?"라고 되물을 수 있다.
- **새 기능** 추가 시 "이 도구로 가능한가"를 먼저 판단할 수 있다.

---

## 1. Next.js 16 (App Router)

### 뭐하는 건가
React 기반 풀스택 프레임워크. **프론트(브라우저 화면) + 백엔드(API) + 서버 사이드 렌더링**을 한 프로젝트 안에서 처리.

### 왜 Next.js인가
- **Server Component** 덕분에 API 라우트 없이도 DB 조회 결과를 HTML에 바로 심을 수 있다 → 로딩 속도↑, 코드 양↓.
- **Vercel과 짝꿍** — Vercel이 Next.js 만든 회사. 배포·Preview·Cron 세팅이 클릭 수준.
- **SEO** — 서버에서 HTML을 완성해 보내서 검색엔진에 그대로 노출. React 단독 SPA는 이게 어렵다.

### 대안이었던 것들
| 대안 | 거절 이유 |
|------|----------|
| **React + 별도 백엔드(Express)** | 프론트·백엔드 2개 레포 관리. 타입 공유 복잡. |
| **Remix** | 생태계가 Next.js보다 작음. 국내 사례 적음. |
| **Nuxt (Vue)** | 팀이 React에 익숙. Vue로 가면 학습비용. |

### 우리가 쓰는 버전
`next@16.1.6`, `react@19.2.3`. ([package.json:22](../../../jidokhae-web/package.json))

### 이걸 만질 때 건드리는 파일
- `next.config.ts` — 빌드·이미지·리다이렉트 설정
- `src/app/` — 라우트 전체
- `src/middleware.ts` — 전역 요청 처리

## 2. Supabase (Postgres + Auth + Storage)

### 뭐하는 건가
**"Firebase의 오픈소스 버전"**이라고 알려져 있지만, 엄밀히는 PostgreSQL 위에 Auth·Storage·Realtime을 얹은 BaaS(Backend-as-a-Service).

우리에겐 3가지 서비스의 묶음:
1. **Postgres DB** — 실제 데이터 저장
2. **Auth** — 카카오 OAuth 연동
3. **Row Level Security (RLS)** — 테이블 행 단위 권한

### 왜 Supabase인가
- **RLS가 핵심** — 로그인한 회원이 **자기 신청 내역만** 조회하도록 DB가 직접 강제한다. 앱 코드에서 실수해도 DB가 막아준다. 다른 BaaS로는 이 수준의 안전망이 어렵다.
- **SQL 쓸 수 있음** — Firebase는 자체 DSL. Supabase는 표준 Postgres → 개발자 뇌에 맞음, 시장 인력 확보 쉬움.
- **한국 카카오 OAuth 지원** — 공식 provider에 카카오 포함.

### 대안이었던 것들
| 대안 | 거절 이유 |
|------|----------|
| **Firebase** | NoSQL 구조가 "회원-모임-신청" 관계 표현에 불편. RLS 같은 행 단위 권한 약함. |
| **AWS RDS + 직접 Auth 구현** | 1인 운영에 부담. OAuth 구현, 세션 관리, 이메일 인증 모두 자가 구축. |
| **Hasura** | Postgres + GraphQL이지만 셀프 호스팅 복잡. |

### 핵심 보안 장치
| 장치 | 위치 | 역할 |
|------|------|------|
| RLS 정책 | `supabase/migration.sql` CREATE POLICY 구문 (L87~) | 로그인 본인 데이터만 조회 |
| `is_admin()` / `is_editor_or_admin()` 함수 | `migration.sql` L73, L292 | 관리자 권한 체크 |
| SECURITY DEFINER | DB 함수들 | 무한 재귀 방지 + 권한 상승 |

### 이걸 만질 때 건드리는 파일
- `supabase/migration-*.sql` — 스키마 수정
- [src/lib/supabase/server.ts](../../../jidokhae-web/src/lib/supabase/server.ts), [client.ts](../../../jidokhae-web/src/lib/supabase/client.ts), [admin.ts](../../../jidokhae-web/src/lib/supabase/admin.ts) — 3가지 클라이언트
- `src/types/*.ts` — 수동 타입 (Supabase 자동 생성 안 씀)

## 3. TossPayments

### 뭐하는 건가
한국 결제 대행사(PG). 카드사 여러 곳을 우리가 직접 계약하지 않고 **토스 한 곳과만 계약**하면 모든 카드 결제 처리.

### 왜 TossPayments인가
- **국내 최저 수준 수수료**: 중소사업자 기준 약 2.x%.
- **개발자 문서 품질**: 한국 PG 중 압도적. SDK가 TypeScript 지원.
- **심사 통과 시** 카드결제 지원. 심사 중에는 **계좌이체 브릿지** 전환 가능 → 우리 프로젝트는 이 전환이 중요했다.

### 대안이었던 것들
| 대안 | 거절 이유 |
|------|----------|
| **KG이니시스 / 나이스페이** | 수수료 높음, 개발 경험 떨어짐. |
| **Stripe** | 국내 카드 결제 미지원. |
| **카카오페이 단독** | 카드 결제 커버 못 함. 제휴가 아니면 단독 결제만. |

### 주의 사항
- **redirect-only**: 팝업/인라인 결제 방식 사용 안 함. 결제 페이지로 **전환** 후 돌아옴. 이유는 Part 4 참조.
- **Webhook 백업 필수**: 리다이렉트 도중 사용자가 뒤로가기 눌러도 결제 상태 추적 가능하려면 webhook 병행.

### 이걸 만질 때 건드리는 파일
- [src/lib/tosspayments.ts](../../../jidokhae-web/src/lib/tosspayments.ts) — API 호출 래퍼
- [src/lib/payment.ts](../../../jidokhae-web/src/lib/payment.ts) — 결제 확정 비즈니스 로직
- `src/app/api/registrations/confirm/route.ts` — 결제 확인 API
- `src/app/api/webhooks/tosspayments/route.ts` — 웹훅 수신

## 4. Tailwind CSS v4

### 뭐하는 건가
**유틸리티 클래스 기반 CSS 프레임워크**. `<div class="bg-white p-4 rounded">` 식으로 작은 클래스를 조합해 스타일링.

### 왜 Tailwind인가
- **CSS 파일 따로 관리 안 해도 됨** — JSX 안에 클래스로 끝.
- **디자인 토큰을 한 곳에** — 색상·폰트를 `@theme inline` 블록에 정의하면 전역 공유.
- **v4 특징**: JS 설정 파일(`tailwind.config.ts`) 제거. 전부 CSS 안에서 설정.

### 대안이었던 것들
| 대안 | 거절 이유 |
|------|----------|
| **CSS Modules** | 컴포넌트마다 `.module.css` 파일 생성, 파일 폭증. |
| **styled-components** | 런타임 CSS 생성 → 초기 로딩 느려짐. Next.js 16과 궁합도 애매. |
| **Bootstrap** | 기성 디자인에 묶임. 커스텀 어려움. |

### 우리의 디자인 시스템
"Editorial Organic" — Deep Forest Green + Warm Terracotta + Warm Gray + Ivory.
자세한 토큰: [jidokhae-web/DESIGN_TOKENS.md](../../../jidokhae-web/DESIGN_TOKENS.md).

### 이걸 만질 때 건드리는 파일
- `src/app/globals.css` — `@theme inline` 블록에 토큰 정의
- `DESIGN_TOKENS.md` — 토큰 레퍼런스
- 개별 컴포넌트의 `className` 속성

## 5. Vercel

### 뭐하는 건가
Next.js를 만든 회사의 **호스팅 플랫폼**. `git push`하면 자동 배포. PR마다 Preview 환경 자동 생성. Cron까지 내장.

### 왜 Vercel인가
- **Next.js와 1급 호환** — Edge 함수, 이미지 최적화, ISR 등이 기본 동작.
- **Preview 자동화** — 각 PR의 미리보기 URL이 자동. 수동 배포 없이 검증.
- **Cron 내장** — 별도 크론 서버 없이 `vercel.json`에 스케줄만 적으면 끝.
- **무료 티어로도 250명 규모는 충분**.

### 대안이었던 것들
| 대안 | 거절 이유 |
|------|----------|
| **AWS (EC2 + CloudFront)** | 1인 운영엔 과도. 초기 세팅·유지보수 시간 소모. |
| **Cloudflare Pages** | Next.js 호환성이 Vercel보다 떨어짐. |
| **Netlify** | Vercel만큼 Next.js와 밀접하지 않음. Cron 별도 세팅. |

### Preview 환경 주의
**Preview는 prod Supabase에 그대로 연결**. 별도 staging DB 없다 — 테스트 모임 만들면 실제 DB에 들어간다. 테스트 후 반드시 삭제.

### 이걸 만질 때 건드리는 파일
- [vercel.json](../../../jidokhae-web/vercel.json) — Cron 스케줄
- Vercel Dashboard — 환경변수 관리
- `.env.example` — 필요한 env 목록의 템플릿

## 6. Solapi (알림톡)

### 뭐하는 건가
카카오 알림톡 발송 대행사. 카카오가 직접 개발자에게 API를 열어주지 않기 때문에, **허가받은 대행사**를 통해 발송.

### 왜 Solapi인가
- **카카오 알림톡 공식 파트너** — 템플릿 사전 심사 후 발송.
- **Node.js SDK** — 타입 지원 있음.
- **가격 안정**: 건당 ~9원 내외.

### 대안이었던 것들
| 대안 | 거절 이유 |
|------|----------|
| **카카오 직접 연동** | 비즈 채널은 직접 API 거의 불가. |
| **Aligo / 뿌리오** | 사용 경험 더 낮음. |
| **문자(SMS)로 대체** | 건당 비용 3배. 카톡 도달률이 더 높음. |

### 현재 5개 템플릿
| 용도 | 트리거 |
|------|-------|
| 신청 확인 | 결제 성공 직후 |
| 모임 전날 리마인드 | Cron (KST 19:00) |
| 대기 신청 확인 | 대기 INSERT 직후 |
| 대기 승격 확정 | 누군가 취소 → 자동 승격 |
| 미승격 자동 환불 | Cron (KST 18:30), 모임 전날 |

### 이걸 만질 때 건드리는 파일
- [src/lib/solapi.ts](../../../jidokhae-web/src/lib/solapi.ts) — SDK 래퍼
- [src/lib/notification.ts](../../../jidokhae-web/src/lib/notification.ts) — 발송 + 이력 기록
- Vercel 환경변수 — 5개 Template ID + API 키

## 7. 조연급: Vitest / ESLint / Playwright

완성도를 위해 짧게.

| 도구 | 역할 | 언제 쓰나 |
|------|------|----------|
| **Vitest** | 단위 테스트 러너 | `npm test` — kst, refund 함수 테스트 |
| **ESLint** | 코드 스타일 검사 | `npm run lint` — 커밋 전 자동 체크 |
| **Playwright** | 브라우저 자동화 | `npm run screenshot` — UI 캡처 전용. E2E 테스트는 아직 안 씀 |

E2E 테스트가 없는 이유는 Part 5 "수용·거절 판단기준"에서 다룬다 (MVP 단계의 의도적 결정).

## 에이전트가 "X로 바꾸자" 제안할 때 던질 3가지 질문

1. **"현재 제약(1인 운영 / 한국 250명 / 카드 심사)에 맞나?"**
   스타트업 흔한 함정 — Stripe이 좋다는 이유로 국내 카드 커버를 놓치는 식.
2. **"이전한 뒤의 유지보수 비용은?"**
   신규 도구가 더 좋아도, 익숙하지 않으면 장기 부담.
3. **"지금 해결하려는 문제가 정말 그 도구로 풀리나?"**
   (예: "느려서 DB를 MongoDB로" → 실제 병목은 인덱스 부족일 수 있다.)

---

> **기억할 것**
> 1. Next.js + Supabase + TossPayments + Vercel + Solapi + Tailwind = 우리 스택. 각각 **"왜 골랐는가"** 가 있다.
> 2. 스택 교체 제안 시 **"현재 제약"** 을 먼저 꺼내 검증. 유행 따라 바꾸면 지속 비용이 커진다.
> 3. 스택은 "도구"일 뿐 **핵심 자산은 DB 스키마와 비즈니스 로직**(`src/lib/`). 도구가 바뀌어도 이게 안 깨지면 이전 가능.
