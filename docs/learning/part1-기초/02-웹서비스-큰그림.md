# 02. 웹서비스 큰그림

> **이 챕터에서 배우는 것**
> 1. 웹서비스는 **3개의 역할**로 이루어진다 — 브라우저 / 서버 / DB
> 2. 각 역할이 JIDOKHAE의 **어느 파일**에 대응되는가
> 3. "신청 버튼"을 눌렀을 때 이 3개가 **어떻게 대화**하는가

---

## 1. 가장 큰 그림 — 3가지 역할

우리 서비스는 본질적으로 **3층 건물**이다.

```
┌─────────────────────────────────────┐
│  브라우저 (사용자 손안에 있는 것)          │
│  = 진열대, 메뉴판, 버튼                  │
└─────────────────┬───────────────────┘
                  │ HTTP 요청/응답
┌─────────────────┴───────────────────┐
│  서버 (Vercel에서 돌아감)                │
│  = 주문 받고 일 처리하는 점원             │
└─────────────────┬───────────────────┘
                  │ SQL 쿼리
┌─────────────────┴───────────────────┐
│  DB (Supabase Postgres)              │
│  = 창고, 장부                          │
└─────────────────────────────────────┘
```

### 각 층이 하는 일

| 층 | 비유 | 하는 일 |
|---|------|--------|
| 브라우저 | 진열대 + 버튼 | 사용자에게 모임 카드 보여주기, 클릭 받기 |
| 서버 | 점원 | 요청 받아서 판단, 결제 확인, DB 읽기/쓰기 |
| DB | 창고 | 회원·모임·신청 데이터 영구 저장, 정원 락 |

**중요**: 브라우저는 DB에 **직접** 못 닿는다. 반드시 서버를 거친다.
(보안상 당연한 것 — 고객이 창고 금고에 직접 손대면 안 되듯이.)

## 2. JIDOKHAE의 3층 지도

우리 프로젝트의 파일들이 3층 중 어디에 속하는지.

### 브라우저 층 (사용자 눈에 보이는 것)

| 경로 | 하는 일 |
|------|--------|
| [src/components/meetings/MeetingCard.tsx](../../../jidokhae-web/src/components/meetings/MeetingCard.tsx) | 모임 카드 한 장 그리기 |
| [src/components/meetings/MeetingActionButton.tsx](../../../jidokhae-web/src/components/meetings/MeetingActionButton.tsx) | 신청/대기/취소 버튼 |
| `src/app/(main)/**/page.tsx` | 각 화면 전체 레이아웃 |
| `src/app/globals.css` | 색상·폰트 등 디자인 토큰 |

맨 위에 `'use client'`가 붙은 파일 = 브라우저에서 실행되는 부분.

### 서버 층 (Vercel에서 돌아감)

| 경로 | 하는 일 |
|------|--------|
| [src/middleware.ts](../../../jidokhae-web/src/middleware.ts) | 모든 요청에서 로그인 상태 확인 |
| `src/app/api/**/route.ts` | API 엔드포인트(결제 확인, 취소, 크론 등) |
| [src/lib/payment.ts](../../../jidokhae-web/src/lib/payment.ts) | 결제 확정 로직 |
| [src/lib/cancel.ts](../../../jidokhae-web/src/lib/cancel.ts) | 취소/환불 로직 |
| [src/lib/tosspayments.ts](../../../jidokhae-web/src/lib/tosspayments.ts) | TossPayments API 호출 래퍼 |
| [src/lib/solapi.ts](../../../jidokhae-web/src/lib/solapi.ts) | 알림톡 발송 래퍼 |

Server Component인 `page.tsx`도 이쪽이다 — 서버에서 DB 조회 후 HTML을 만들어 브라우저로 보낸다.

### DB 층 (Supabase에 있음)

| 경로 | 하는 일 |
|------|--------|
| [supabase/migration.sql](../../../jidokhae-web/supabase/migration.sql) | 테이블·인덱스·함수·정책의 설계도 |
| `supabase/migration-phase3-m7.sql` | Phase 3에서 추가된 컬럼·테이블 |
| DB 안의 `confirm_registration()` 함수 | 결제 확정 시 원자적 정원 체크 |
| DB 안의 `is_admin()` 함수 | 권한 체크 (RLS 정책이 호출) |

이 파일들은 Supabase SQL Editor에 **붙여넣어 실행**된 순간 실제로 동작한다. 파일을 수정해도 실행 안 하면 DB엔 반영 안 됨.

## 3. 사용자가 신청 버튼을 누르면 — 시퀀스 투어

모임 신청 한 건에 3개 층이 어떻게 대화하는지 따라가보자.

```
[회원]            [브라우저]              [서버]                 [TossPayments]          [DB]
  │                   │                     │                        │                    │
  │ ① "신청하기" 클릭  │                     │                        │                    │
  │─────────────────▶│                     │                        │                    │
  │                   │ ② TossPayments SDK                            │                    │
  │                   │    결제 창 띄우기    │                        │                    │
  │                   │────────────────────────────────────────────▶│                    │
  │                   │                     │                        │                    │
  │ ③ 카드 정보 입력   │                     │                        │                    │
  │─────────────────────────────────────────────────────────────────▶│                    │
  │                   │                     │                        │                    │
  │                   │ ④ 결제 성공, 우리 페이지로 redirect              │                    │
  │◀───────────────────────────────────────────────────────────────│                    │
  │                   │                     │                        │                    │
  │                   │ ⑤ /api/registrations/confirm 호출              │                    │
  │                   │────────────────────▶│                        │                    │
  │                   │                     │                        │                    │
  │                   │                     │ ⑥ confirmPayment (돈 실제 이동)                │
  │                   │                     │──────────────────────▶│                    │
  │                   │                     │                        │                    │
  │                   │                     │ ⑦ confirm_registration() RPC 호출              │
  │                   │                     │─────────────────────────────────────────────▶│
  │                   │                     │                        │     ⑧ FOR UPDATE 락  │
  │                   │                     │                        │        정원 체크      │
  │                   │                     │                        │        INSERT        │
  │                   │                     │ ⑨ 'success' 응답        │                    │
  │                   │                     │◀─────────────────────────────────────────────│
  │                   │                     │                        │                    │
  │                   │                     │ ⑩ 알림톡 발송 (fire-and-forget)                │
  │                   │                     │────────▶ Solapi                              │
  │                   │                     │                        │                    │
  │                   │ ⑪ "신청 완료" 화면    │                        │                    │
  │                   │◀────────────────────│                        │                    │
  │ ⑫ 알림톡 카톡 도착  │                     │                        │                    │
  │◀──────────────────────────────────────── (카카오가 배달)                                │
```

### 이 그림에서 읽어낼 3가지 핵심

**① 브라우저는 DB와 직접 대화 못 한다.**
⑦의 "confirm_registration() 호출"은 서버(⑤)만 할 수 있다. 브라우저에서 바로 DB에 INSERT 시도할 수 없게 설계되어 있다([RLS](01-용어사전.md#27-rls-row-level-security) 덕분).

**② 돈은 ⑥에서 실제로 움직인다.**
그 이전까지는 "결제 의도"만 있는 상태. ⑥ 직후에 DB INSERT(⑦)가 실패하면 돈은 이미 나갔는데 신청 기록이 없는 상황이 된다 → 그래서 ⑥ 직후 DB 실패 시 **자동 환불**로 롤백한다.

**③ 알림톡(⑩)은 "부가 기능"이다.**
알림톡이 실패해도 ⑨의 신청 확정은 그대로 유지된다. 알림 실패가 결제 흐름을 막으면 안 되기 때문(`fire-and-forget` 패턴). 이건 Part 4의 설계 결정 챕터에서 다룬다.

## 4. 왜 이렇게 3층으로 나눴나

### 짧게: **보안 + 역할 분리 + 확장성**

- **보안**: 브라우저는 사용자가 열어볼 수 있다. 브라우저 코드에 `service_role key`를 박아두면 하루 만에 유출된다. 그래서 DB 직접 접근은 서버만.
- **역할 분리**: 브라우저는 "보여주기"만, 서버는 "판단·실행", DB는 "기록·정합성". 한쪽이 바뀌어도 다른 쪽 영향 최소화.
- **확장성**: 사용자 1만 명 되면 브라우저는 각자 컴퓨터에서 돌고, 서버만 여러 대로 늘리면 된다. 서비스가 커져도 같은 구조.

### 조금 길게: "왜 Next.js는 프론트+서버를 한 레포에?"

전통적으로 프론트/백엔드 레포가 분리되던 것과 달리, Next.js는 한 레포에서 양쪽을 한다. 장점:

1. **타입 공유** — `types/meeting.ts`를 프론트·서버·테스트가 같이 쓴다.
2. **Server Component** — "DB 조회 → HTML 만들기"를 한 파일 안에서 처리. API 라우트 없이도 서버 로직 실행.
3. **배포 단순화** — 프론트 서버 1개만 관리.

단점은 다음 챕터에서 다룬다.

## 5. 실전 팁 — 코드를 볼 때 "어느 층인가" 먼저 물어라

낯선 파일을 열었을 때 이 순서로 확인하면 덜 당황한다.

1. 파일 맨 위에 `'use client'`가 있나? → **브라우저 층**
2. 경로가 `src/app/api/.../route.ts`인가? → **서버 층 (API 라우트)**
3. 경로가 `src/lib/`인가? → **서버 층 (로직 모음)**
4. 경로가 `src/middleware.ts`인가? → **서버 층 (모든 요청 공통 처리)**
5. 경로가 `supabase/*.sql`인가? → **DB 층 (설계도)**
6. 위에 어떤 것에도 해당 안 되고 `page.tsx`인가? → 기본적으로 **서버 층 Server Component**

이 분류만 해도 파일의 절반은 이해한 셈이다.

---

> **기억할 것**
> 1. 웹서비스는 **브라우저 ↔ 서버 ↔ DB** 3층 구조. 브라우저는 DB에 직접 못 닿는다.
> 2. JIDOKHAE에서 브라우저 = `src/components/`, 서버 = `src/app/api/` + `src/lib/`, DB = `supabase/*.sql`.
> 3. "돈이 움직이는 지점"과 "DB가 변하는 지점"을 찾으면, 어느 파일이 중요한지 바로 보인다.
