## 왜 이 책을 쓰게 됐는가

프로젝트가 MVP(M6 출시)를 넘어 Phase 3까지 오면서, 총괄 파트너 입장에서 "모르는 게 점점 많아지는" 경험이 누적되고 있다.

- `/full-scan` 리포트를 펼쳤는데, 7명의 에이전트가 만든 보고서의 **용어부터 낯설다**.
- "이 파일이 왜 여기 있지?" — **파일 구조**가 눈에 안 들어온다.
- Claude가 "이건 idempotency 보장이 필요합니다"라고 말하는데, **"그게 왜 중요한지"** 에 대한 감이 없어 끄덕이기만 한다.

이 상태로는 에이전트의 제안을 **비판적으로 검토**할 수 없다. 전부 수용하거나, 전부 거절하거나 둘 중 하나가 된다.

→ 목표: **에이전트의 제안에 "왜?"라고 되물을 수 있는 수준**을 만들기.

## 이 책이 다루는 것 / 다루지 않는 것

### 다루는 것

- 이 프로젝트(`jidokhae-web/`)에서 실제 쓰이는 용어와 패턴
- 코드를 **읽는 법** (읽을 때 어디를 먼저 볼지, 뭘 무시할지)
- 설계 결정의 **배경**과 **대안 비교**
- `/full-scan` 같은 리포트를 읽고 **판단**하는 기준

### 다루지 않는 것

- 코딩 튜토리얼 — "JavaScript 문법" 같은 것은 없다.
- 범용 웹 개발 지식 — 우리 프로젝트 맥락에 고정한다.
- 최신 트렌드 — 2026-04-24 시점의 이 코드베이스에 맞춘다.

## 읽는 법 3가지

### 1. 용어부터 (권장 시작점)

[01. 용어사전](part1-기초/01-용어사전.md)을 **쭉 훑는다.** 처음에는 외우지 말고 비유만 눈에 담기. 나중에 리포트에서 해당 용어를 만나면 돌아와서 확인.

### 2. 흐름부터 (코드가 궁금할 때)

Part 3의 "흐름 투어" 챕터를 먼저 본다. **버튼 클릭 → 내부에서 벌어지는 일**이 시간 순서로 그려져 있어서, 맥락으로 용어를 익히게 된다.

### 3. 리포트 해독부터 (당장 실용적으로 써먹고 싶을 때)

Part 5가 완성되면, 실제 `/full-scan` 리포트 발췌본 옆에 **"이 지적이 뭘 의미하는가"** 해설이 붙어 있다. 여기부터 읽고 거꾸로 Part 1~4로 내려와도 된다.

## 막힐 때 Claude에게 물어보는 법

이 책 자체가 읽히는 중에 불완전할 수 있다. 막히면 다음 패턴으로 묻자.

| 상황 | 프롬프트 예시 |
|------|--------------|
| 비유가 어렵다 | "01 용어사전 [용어] 부분, 다른 비유로 다시 설명해줘" |
| 챕터가 짧아서 이해 안 된다 | "02 큰그림 챕터를 예시 3개 더 붙여서 풀어줘" |
| 실제 코드가 보고 싶다 | "이 챕터에서 말한 [파일명] 실제 내용 20줄만 보여주고 해설해줘" |
| 챕터가 너무 기술적이다 | "비전공자가 알아듣게 1/3 분량으로 다시 써줘" |
| 보강 요청 | "이 챕터에 내가 체크한 부분을 더 자세히 설명해서 업데이트해줘" |

Claude는 이 책을 **읽고 업데이트할 수 있다.** 막히면 말하라 — 그게 책을 개선하는 유일한 방법이다.

---

## 이 책의 업데이트 규칙

- 용어 추가: Part 1의 용어사전에만 추가.
- 흐름이 바뀌면: Part 3의 해당 흐름 챕터만 수정.
- 설계 결정이 번복되면: Part 4의 해당 결정 챕터에 "**변경 이력**" 섹션 추가.
- 폐기된 내용: 지우지 말고 "(2026-XX-XX 폐기)" 마크만 달기.
