# JIDOKHAE MVP Milestones

> 이 문서는 core 문서(서비스 개요, PRD, 기술 스택 & 시스템 구조)를 분석하여 수립한 **MVP 개발 최상위 전략**입니다.
> 각 Milestone은 독립적으로 테스트 가능한 산출물을 가지며, 의존성 순서에 따라 배치되어 있습니다.

---

## Milestone 의존성 흐름

```
M1 → M2 → M3 → M4 → M5 → M6
```

> M3의 회원 기능과 운영자 기능은 기술 의존성(운영자 UI가 모임 상세 페이지에 의존)으로 인해 직렬 처리됩니다.

| 순서 | Milestone | 의존 대상 | 산출물 |
|:---:|-----------|----------|--------|
| M1 | 프로젝트 기반 구축 | — | 배포 가능한 빈 Next.js 앱 + Supabase DB |
| M2 | 인증 (카카오 로그인) | M1 | 로그인/로그아웃 + 사용자 프로필 자동 생성 |
| M3 | 모임 일정 조회 + 운영자 CRUD | M2 | 회원: 일정 목록/상세 조회, 운영자: 모임 생성/수정/삭제(환불 제외) |
| M4 | 결제 + 신청 | M3 | 모임 신청/결제 → 신청 확정 전체 플로우 |
| M5 | 취소 + 환불 | M4 | 셀프 취소/자동 환불 + 모임 삭제 시 일괄 환불 |
| M6 | 통합 검증 + 출시 준비 | M5 | 전체 E2E 검증 완료, 프로덕션 배포 |

---

## M1. 프로젝트 기반 구축

### 목표
개발의 토대가 되는 프로젝트 뼈대와 인프라를 세팅한다. 이후 모든 Milestone이 이 기반 위에서 진행된다.

### 산출물
- [x] Next.js (App Router, TypeScript) 프로젝트 초기화
- [x] Vercel 연동 + GitHub push 시 자동 배포 확인
- [x] Supabase 프로젝트 생성 + DB 스키마 구축
  - `profiles`, `meetings`, `registrations` 테이블
  - RLS 정책 설정 (테이블별 member/admin 권한)
  - DB Function: `confirm_registration` (정원 체크 + 원자적 INSERT)
  - DB Function: 모임별 confirmed 수 조회 (SECURITY DEFINER)
  - DB Trigger: `auth.users` INSERT 시 `profiles` 자동 생성
- [x] 환경 변수 설정 (Supabase URL/Keys, 포트원 API Key 등)
- [x] Supabase Client 3종 유틸리티 (Server용/Client용 anon key, Admin용 service_role key)
- [x] 모바일 우선 반응형 레이아웃 기본 구조 (하단 탭: 모임 일정 / 내 신청)
- [x] 글로벌 스타일 시스템 (디자인 토큰, 타이포그래피)

### 완료 기준
- Vercel에 빈 앱이 배포되어 접속 가능
- Supabase DB에 3개 테이블 + RLS + Function + Trigger 생성 확인
- 모바일/PC에서 기본 레이아웃이 렌더링됨

## M2. 인증 (카카오 로그인)

### 목표
카카오 소셜 로그인으로 회원을 식별하고, 로그인 상태에 따라 페이지 접근을 제어한다.

### 의존성
- **M1 필요**: Supabase Auth 설정, profiles 테이블, DB Trigger

### 산출물
- [x] 카카오 로그인 화면 (카카오 로그인 버튼)
- [x] Supabase Auth ↔ 카카오 OAuth 연동
- [x] 로그인 성공 시 → 모임 일정 목록(메인)으로 이동
- [x] 재방문 시 세션 유지 (바로 메인 진입)
- [x] 로그아웃 기능
- [x] profiles 자동 생성 확인 (DB Trigger 동작)
- [x] 운영자(admin) role 수동 설정 및 확인
- [x] 미인증 사용자 → 로그인 화면으로 리다이렉트 (미들웨어)

### 완료 기준
- 카카오 계정으로 로그인/로그아웃이 동작
- 로그인 시 `profiles` 레코드가 자동 생성됨
- 미인증 접근 시 로그인 화면으로 리다이렉트됨
- admin role 사용자와 member 사용자 구분 가능

## M3. 모임 일정 조회 + 운영자 CRUD

### 목표
회원이 모임 일정을 한눈에 확인하고, 운영자가 모임을 생성/수정/삭제할 수 있다. 결제와 환불을 제외한 모임 관리의 뼈대를 완성한다.

### 의존성
- **M2 필요**: 로그인된 사용자(auth.uid), member/admin 역할 구분

### 산출물

**회원 기능:**
- [ ] 모임 일정 목록 화면 (메인)
  - 다가오는 모임 표시 (오늘 포함 이후, status=active)
  - 남은 자리 계산 + "마감" 뱃지
  - "신청완료" 표시 (confirmed 기준, cancelled 무시)
  - 날짜 오름차순 정렬
  - "참여하고 싶게 생긴" 시각적 디자인 (카드 UI)
- [ ] 모임 상세 페이지
  - 날짜, 시간, 장소, 남은 자리, 참가비 표시
  - 버튼 분기 5종 (confirmed 유무 × 모임 시점)
  - 단, 이 단계에서 "신청하기" 버튼은 M4에서 연결

**운영자 기능:**
- [ ] 운영자 전용 모임 목록 (과거 + deleting 포함, deleted 제외, 날짜 내림차순)
- [ ] 모임 생성 폼 (모임명, 날짜, 시간, 장소, 정원, 참가비)
- [ ] 모임 수정 기능
- [ ] 모임 삭제 기능 (신청자가 없는 경우만 — 환불 로직은 M5에서)
- [ ] 신청자 목록 UI 틀 (데이터는 M4 이후)

### 완료 기준
- 회원: 모임 일정 목록이 3초 내 로딩, 마감/신청완료 상태 정확히 표시
- 운영자: 모임 생성 → 목록에 즉시 반영, 수정/삭제 동작 확인
- 모바일에서 카드 UI가 매력적으로 렌더링됨

## M4. 결제 + 신청

### 목표
핵심 비즈니스 플로우를 완성한다: **모임 선택 → 결제 → 신청 확정**. 3클릭 이내에 결제 화면 진입.

### 의존성
- **M3 필요**: 모임 상세 페이지, "신청하기" 버튼 UI
- **외부 의존성**: 포트원 V2 API 연동, PG사(토스페이먼츠/KCP) 심사 승인

### 산출물
- [ ] 포트원 SDK 연동 (프론트엔드 — 결제창 호출)
- [ ] 결제 검증 API Route (`/api/registrations/confirm`)
  - 포트원 REST API로 결제 검증
  - DB Function(`confirm_registration`) 호출 → 정원 체크 + 원자적 INSERT
  - 정원 초과 시 자동 환불 + "마감" 응답
- [ ] 신청 확정 화면 ("신청이 완료되었습니다" + 모임 정보 요약)
- [ ] 결제 실패 처리 (모임 상세로 복귀 + 안내 메시지)
- [ ] 내 신청 내역 화면
  - 신청한 모임 목록 + 상태 뱃지 (신청완료 / 참여 완료 / 취소됨)
  - 과거 이력 포함, 날짜 내림차순
- [ ] 모임 상세 버튼 분기 완전 연결 (신청하기 → 결제 플로우)
- [ ] 신청자 목록 데이터 연동 (운영자: 이름, 결제 상태, 결제 일시)

### 완료 기준
- 회원이 모임 선택 → 결제 완료까지 3클릭 이내
- 결제 완료 = `registrations` INSERT 확인 (status: confirmed)
- 동시 신청 시 정원 초과분은 자동 환불 처리
- 내 신청 내역에서 신청 건 확인 가능
- 운영자가 신청자 목록 + 결제 상태 확인 가능

## M5. 취소 + 환불

### 목표
회원 셀프 취소/자동 환불과 운영자 모임 삭제 시 일괄 환불을 구현하여, MVP 핵심 흐름(일정 → 신청/결제 → 취소/환불)을 완성한다.

### 의존성
- **M4 필요**: 결제 완료된 신청 레코드(registrations), 포트원 연동

### 산출물

**회원 취소:**
- [ ] 취소하기 버튼 동작 (모임 당일까지 가능, 다음날부터 취소 불가)
- [ ] 환불 규정 안내 표시 (3일 전 100% / 2일 전 50% / 이후 0%)
- [ ] 취소 확정 확인 다이얼로그 ("취소를 확정하시겠습니까?")
- [ ] 취소/환불 API Route (`/api/registrations/cancel`)
  - KST 기준 환불 금액 자동 계산
  - 포트원 환불 API 호출 (0원이면 생략)
  - `registrations` 상태 업데이트 (cancelled, user_cancelled)
- [ ] 취소 완료 화면 (환불 예정 금액 + 소요 시간 안내)
- [ ] "단무지에게 1대1톡" 문의 링크
- [ ] 취소 후 재신청 가능 확인 (자리 남아있을 시)

**운영자 모임 삭제 + 일괄 환불:**
- [ ] 삭제 시 "신청자 N명에게 환불됩니다" 확인 절차
- [ ] 모임 삭제 API Route (`/api/meetings/[id]/delete`)
  - `meetings.status` → `deleting`
  - confirmed 신청자 전원 병렬 환불 (`Promise.allSettled`)
  - 전원 성공 → `deleted` / 실패 건 → `deleting` 유지
- [ ] 환불 실패 건 재시도 흐름 (운영자 화면)
- [ ] deleting 상태 모임 "환불 미처리" 표시

### 완료 기준
- 회원: 취소 → 환불 금액이 규정에 따라 정확히 계산/처리됨
- 회원: 환불 불가 기간에도 취소 가능 (0원 안내 포함)
- 운영자: 모임 삭제 시 일괄 환불 동작, 실패 건 재시도 가능
- 모임 날짜 다음날부터 취소 불가 확인

## M6. 통합 검증 + 출시 준비

### 목표
전체 MVP 흐름을 E2E로 검증하고, 프로덕션 환경에서 실제 운영 가능한 상태로 준비한다.

### 의존성
- **M5 필요**: 모든 기능 구현 완료

### 산출물
- [ ] E2E 테스트 시나리오 수행
  - 회원: 로그인 → 일정 확인 → 신청/결제 → 내역 확인 → 취소/환불
  - 운영자: 모임 생성 → 신청자 확인 → 모임 수정 → 모임 삭제(일괄 환불)
  - 엣지 케이스: 동시 신청(정원 초과), 결제 실패, 환불 실패 재시도, 재신청
- [ ] 프로덕션 환경 설정
  - Supabase 프로덕션 프로젝트 (또는 동일 프로젝트 확인)
  - 포트원 실 결제 모드 전환 + PG사 연동 확인
  - Vercel 프로덕션 도메인 설정
  - 환경 변수 프로덕션 값 세팅
- [ ] 운영자(영탁) 계정 admin 설정
- [ ] 모바일 브라우저 호환성 검증 (카카오톡 인앱 브라우저 포함)
- [ ] 성능 확인 (목록 3초 이내 로딩)
- [ ] 버그 수정 + 최종 QA
- [ ] 카카오톡 오픈채팅방에 서비스 링크 공유 준비

### 완료 기준 (= MVP 출시 기준)
- ✅ 회원이 카카오 로그인으로 접속할 수 있다
- ✅ 모임 일정 목록이 보인다 (마감 상태 포함)
- ✅ 모임을 선택하고 결제까지 완료하면 신청이 확정된다
- ✅ 내 신청 내역을 확인할 수 있다
- ✅ 회원이 직접 취소할 수 있고, 환불 규정에 따라 자동 환불된다
- ✅ 운영자가 모임을 생성/수정/삭제할 수 있다
- ✅ 운영자가 신청자 목록과 결제/취소 상태를 확인할 수 있다
