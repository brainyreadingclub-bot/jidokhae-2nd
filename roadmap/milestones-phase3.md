# JIDOKHAE Phase 3 Milestones

> 이 문서는 `roadmap/phase3-requirements.md`(Phase 3 통합 요구사항, 2026-04-16)를 분석하여 수립한 **Phase 3 개발 최상위 전략**입니다.
> 각 Milestone은 독립적으로 테스트 가능한 산출물을 가지며, 의존성 순서에 따라 배치되어 있습니다.
> 기존 `milestones.md`(MVP M1~M6 기록)와 분리되어 있으며, Phase 4+는 별도 파일(`milestones-phaseN.md`)로 추가합니다.

---

## 설계 원칙

1. **수직 슬라이싱(기능 end-to-end)** — 레이어별(DB→관리자→회원→UI) 수평 분할이 아닌, 각 Milestone 끝에 독립적 가치가 배포되는 기능 단위 묶음.
2. **MVP 구축 방식** — 운영자가 데이터를 입력할 수 있어야 회원이 볼 수 있으므로, 관리자 CMS를 선행하고 회원 노출을 뒤에 묶는다.
3. **같은 파일 반복 수정 회피 (교훈 L-5)** — 홈 탭 뼈대/레이아웃은 M7에 몰고, 이후 Milestone은 "빈 칸 채우기" 방식.
4. **UI 작업 전 `/frontend-design` 선행 (필수)** — 신규 화면/섹션/컴포넌트 구성이 포함된 모든 Milestone은 코드 작성 전에 `/frontend-design` 스킬을 먼저 실행해 디자인 방향을 확정한다. 덕분에 "전체 리디자인" 같은 후행 통째 개편이 불필요해진다.

## Milestone 의존성 흐름

```
M7 ─┬─→ M8 → M9 ──┐
    └─→ M10 ──────┼─→ M11 → M12
                  (M9 + M10 병렬 가능)
```

**주요 근거:** M10은 WP10-1/10-2/10-3 모두 WP7(사이드바/DB/모임관리 분리)에만 의존. M8의 "사이드바 메뉴 슬롯"은 실제로 M7에서 만들어지므로 M10은 M8 대기 불필요. M9 완료 후 M10을 시작하는 것이 아니라 M8 이후 M9/M10을 병렬 진행하면 Phase 3 전체 기간 단축.

| 순서 | Milestone | 의존 대상 | 산출물 |
|:---:|-----------|----------|--------|
| M7 | 기반 정리 + 레이아웃 전환 | — | 풀스캔 Critical 0 + DB 스키마 확장 + 관리자 데스크톱 전환 + 회원 3탭 뼈대 |
| M8 | 관리자 CMS (배너 + 한 줄) | M7 | 배너 CRUD (Storage) + 한 줄 승인 워크플로우 |
| M9 | 회원 홈 콘텐츠 전면 오픈 | M7, M8 | 홈 4섹션(배너/D-Day/한 줄/소셜 프루프) + 인라인 배너 + 한 줄 제출 |
| M10 | 관리자 심화 (정산 + 회원 생애주기) | M7 (M8/M9와 병렬 가능) | 정산 admin 전용 + 회원 5단계 분석 + 모임 폼 추가 옵션 |
| M11 | 디자인 토큰 통합 + 접근성 정리 | M9, M10 | 기존 화면 하드코딩 제거 + WCAG AA 대비 통과 |
| M12 | 통합 검증 + 배포 | M11 | 풀스캔 재실행 + 실사용자 회귀 + 프로덕션 배포 |

## M7. 기반 정리 + 레이아웃 전환

### 목표
Phase 3 전체 공사를 받칠 토대를 놓는다 — 기존 풀스캔 이슈 제거, DB 확장, 관리자 데스크톱 전환, 회원 3탭 뼈대까지 한 번에 마무리하여 이후 Milestone이 깔끔히 올라갈 발판을 만든다.

### 의존성
- 없음 (Phase 2-4까지의 프로덕션 상태에서 시작)

### UI 선행 필수
사이드바 + 대시보드 + 회원 3탭 네비는 신규 UI이므로 **코드 작성 전 `/frontend-design` 스킬 선행**. 디자인 방향 확정 후 구현.

### 산출물

**풀스캔 이슈 처리:**
- [ ] 배포 차단급 4건: S-1 웹훅 orderId 교차 검증, S-2 paymentKey encodeURIComponent, S-3 payment.ts orderId 검증, S-4 웹훅 RPC 실패 시 환불 처리
- [ ] 인프라 5건: I-1 보안 헤더, I-2 TossPayments fetch 타임아웃 8초, I-3 attendance API try-catch, I-4 Cron waitlist-refund 병렬화, I-5 confirm-transfer 정원 체크
- [ ] 문구/UX 2건: U-1 "입금 후 운영자 확인까지 대기합니다" 삭제, U-2 API 응답 형식 통일(`{ status, message }`)

**DB 마이그레이션:**
- [ ] `meetings` 컬럼 5종 추가: region (TEXT DEFAULT '경주'), is_featured (BOOLEAN DEFAULT false), chat_link (TEXT nullable), reading_link (TEXT nullable), detail_address (TEXT nullable)
- [ ] `banners` 신규 테이블: title, description, link_url, image_url, is_active, display_order
- [ ] `book_quotes` 신규 테이블: book_title, quote_text, submitted_by, status, approved_at
- [ ] RLS 정책: banners (select_active 로그인 유저 + admin_all FOR ALL), book_quotes (select_approved + select_own + select_editor_or_admin + insert + update_editor_or_admin)

**관리자 데스크톱 전환:**
- [ ] 좌측 사이드바 레이아웃 (데스크톱) + 햄버거 메뉴 (모바일)
- [ ] 사이드바 메뉴 7종: 대시보드, 모임 관리, 정산, 회원 관리, 설정, 배너 관리, 한 줄 관리 (배너/한줄은 독립 메뉴로 분리)
- [ ] 대시보드 `/admin` 허브 뼈대: 긴급 알림 + 이번 주 요약 + 핵심 수치 3개 + 바로가기 링크 (상세 수치는 M10)
- [ ] 모임 관리 `/admin/meetings` (필터: 진행중/지난달/전체 + 지역)
- [ ] 운영자 전용 상세 `/admin/meetings/[id]` (신청자 목록 + 입금 토글 + 재정 요약)
- [ ] AdminMeetingCard 링크 `/admin/meetings/[id]`로 재연결
- [ ] 회원 `/meetings/[id]`의 AdminMeetingSection → "관리자 페이지에서 보기" 링크만

**모임 폼 1차 확장:**
- [ ] region 선택 필수 (한글, `VALID_REGIONS` 참조)
- [ ] is_featured 체크박스 (생성/수정 모두)
- [ ] edit 폼 initialValues에 신규 필드 전달

**회원 탭 구조 전환:**
- [ ] 2탭 → 3탭: 홈 `/home` / 모임 일정 `/` / 내 신청 `/my`
- [ ] 기본 랜딩은 모임 일정(`/`)
- [ ] 홈 `/home`은 placeholder 상태 (섹션은 M9에서 채움)

### 완료 기준
- 풀스캔 재실행 시 Critical 0건
- DB 신규 테이블/컬럼 적용 + RLS 수동 검증 통과
- 관리자 사이드바 + 모임 관리 페이지가 기존 기능 회귀 없이 동작
- 회원 3탭 네비 동작, 기존 로그인/신청/내 신청 흐름 정상
- `npm run prelaunch` 통과 (lint + tsc + test + build)

## M8. 관리자 CMS — 배너 + 한 줄

### 목표
운영자가 회원 홈 콘텐츠 데이터를 등록/관리할 수 있게 한다. 회원 노출은 M9이지만, M8 끝나면 관리 도구는 완전히 사용 가능하다.

### 의존성
- **M7 필요**: banners/book_quotes 테이블, Storage 설정, 사이드바 독립 메뉴 슬롯

### UI 선행 필수
배너 관리 폼, 이미지 업로드 UI, 한 줄 승인 워크플로우는 신규 UI이므로 **코드 작성 전 `/frontend-design` 스킬 선행**.

### 산출물

**배너 관리 (사이드바 독립 메뉴):**
- [ ] 배너 목록 + CRUD (생성/수정/삭제)
- [ ] 활성/비활성 토글
- [ ] 이미지 업로드 (Supabase Storage) — URL 직접 입력 아님
- [ ] 삭제 시 Storage 파일 동시 삭제
- [ ] 표시 순서 설정 + "숫자가 작을수록 먼저" 라벨

**한 줄 관리 (사이드바 독립 메뉴):**
- [ ] 탭 구조: 대기 / 승인됨 / 거절됨
- [ ] 승인 / 거절 / 승인 취소(approved → rejected) / 삭제
- [ ] 닉네임 마스킹 미리보기 (첫 글자 + "○님")

### 완료 기준
- 운영자가 배너 2~3건 등록 후 활성/순서 조정 가능
- 운영자가 book_quotes 수동 INSERT 데이터(또는 M9 이후 회원 제출물)를 승인/거절/승인취소/삭제 가능
- Storage 파일 삭제 동기화 수동 검증 (삭제 후 URL 404 확인)
- RLS 검증: 일반 회원이 배너/한줄 관리 페이지 직접 호출 시 차단

## M9. 회원 홈 콘텐츠 전면 오픈

### 목표
회원이 홈에서 Phase 3의 4섹션 + 인라인 배너를 본다. 서비스의 "재방문 이유"를 만드는 핵심 Milestone.

### 의존성
- **M7 필요**: 3탭 구조, meetings 컬럼 확장
- **M8 필요**: 배너/한 줄 관리 데이터

### UI 선행 필수
홈 4섹션(배너, D-Day 입장권, 한 줄, 소셜 프루프) + 인라인 배너 + 한 줄 제출 폼은 모두 신규 UI이며 서비스 첫인상을 결정한다. **섹션별로 `/frontend-design` 스킬 선행**하여 미학/레이아웃/상호작용 확정 후 구현.

### 산출물

**홍보 배너 (홈 최상단):**
- [ ] 활성 배너 1~2개 가로 스와이프 (CSS scroll-snap)
- [ ] 이미지 배너 / 텍스트 배너 형태 분기 렌더
- [ ] **배너 0개일 때: 다가오는 모임 1~2개 카드 fallback**
- [ ] 링크 처리: 외부 URL `<a target="_blank">`, 내부 `<Link>`, **빈값 `<div>`** (페이지 리로드 방지)

**D-Day 입장권:**
- [ ] 오늘/내일 confirmed 모임만 표시 (없으면 비표시)
- [ ] 티켓 스타일 카드 UI
- [ ] 장소 옆 인라인 "지도" 링크 (네이버 지도, detail_address 기반) — 별도 버튼 행 아님
- [ ] 최대 2개 (오늘 먼저, 내일 아래)

**이번 주 한 줄:**
- [ ] 최근 승인 3~5개 세로 표시 (쌓이는 구조)
- [ ] **"나도 한 줄 남기기" 버튼 항상 표시** (타인 글 유무/제출 후에도 유지)
- [ ] 제출 폼: 책 제목 + 구절/감상 (200자 제한)
- [ ] 닉네임 마스킹: 첫 글자 + "○님"
- [ ] 제출 후 book_quotes.status='pending'으로 INSERT, 운영자 승인 대기

**소셜 프루프:**
- [ ] 4개 항목: 합류 티커 / 최근 가입 / 인기 모임 / 참석 TOP
- [ ] **Supabase N:1 JOIN 타입: 단일 객체 (배열 아님) — 크로스체크 L-3 반영**
- [ ] 데이터 없는 항목만 비표시

**인라인 배너 (모임 일정 탭):**
- [ ] featured 모임을 모임 리스트에 삽입
- [ ] 삽입 규칙: 3번째 아이템 뒤에 1개, 최대 1개 (날짜 그룹마다 아님)
- [ ] 모임 3개 미만이면 맨 아래 배치

### 완료 기준
- 홈 4섹션이 각각 데이터 유/무 경계에서 올바르게 렌더
- 한 줄 제출 → 운영자 승인 → 홈 노출 end-to-end 동작 검증
- 인라인 배너가 featured 모임 있을 때만 렌더
- (main) 라우트이므로 Vercel Preview에서 검증 (L-7 준수)
- 가로 스크롤 UI에서 스크롤바 숨김 검증 (L-6 준수)

## M10. 관리자 심화 — 정산 + 회원 생애주기

### 목표
운영자가 정산과 회원을 대시보드 수준으로 관리한다. M9와 병렬 진행 가능.

### 의존성
- **M7 필요**: 사이드바 레이아웃(WP7-3) + DB 스키마(WP7-2) + 모임 관리 분리(WP7-4)
- **M8/M9와 독립 — 병렬 가능** (M10은 M8의 배너/한 줄 데이터나 M9의 회원 홈 섹션에 의존하지 않음)

### UI 선행 필수
정산 화면(월 선택, 연도 추이, 공간별 정산 테이블) + 회원 생애주기 대시보드(5단계, TOP 랭킹, 이탈 추이)는 신규 UI. **코드 작성 전 `/frontend-design` 스킬 선행**.

### 산출물

**정산 고도화:**
- [ ] `/admin/settlements` admin 전용 이동 (editor 접근 불가)
- [ ] 월 선택 + 연도별 추이
- [ ] 정산 요약 + 공간별 정산 + 정산 완료 처리
- [ ] **재정 요약: 취소 건 환불이 순매출에 정확히 반영 — L-1(좌우 행 범위 일치) 적용**

**회원 관리 생애주기 강화:**
- [ ] 5단계 상태 집계: 신규 / 활성 / 휴면 / 이탈 / 미참석 — **단계별 회원 수 + 회원 목록**
- [ ] 가입 → 첫 참여 소요일 (중앙값, 평균)
- [ ] 재참석 전환율 (1회 → 2회)
- [ ] 지역별 활동량 비교
- [ ] 이번 달 참석 TOP 랭킹
- [ ] 이탈 추이 (월별 이탈자 수)

**모임 폼 2차 확장:**
- [ ] 추가 옵션 접힘 섹션: detail_address, chat_link, reading_link

**링크 정리:**
- [ ] AdminMeetingCard 링크 `/admin/meetings/[id]`로 완전 재연결 (M7에서 설정한 뒤 호출부 정리)
- [ ] `/meetings/[id]`의 AdminMeetingSection → "관리자 페이지에서 보기" 링크 유지

### 완료 기준
- 정산 수치가 confirmed 행만 집계하고 취소 환불을 정확히 차감 (수동 검증용 샘플 쿼리 1건)
- 5단계 집계가 기준 날짜 기반으로 재계산됨
- 모임 폼 추가 옵션 저장/로드 정상 (create/edit 양쪽)
- editor 계정으로 `/admin/settlements` 접근 시 차단

## M11. 디자인 토큰 통합 + 접근성 정리

### 목표
M7~M10에서 각 Milestone마다 `/frontend-design` 선행으로 신규 화면 품질은 이미 확보됨. M11은 **기존 화면(MVP + Phase 2 산출물)**의 하드코딩된 토큰을 통합하고 WCAG AA 대비 기준 미달 부분을 정리한다. "전체 리디자인"이 아니라 **품질 정리**.

### 범위 확정
- ✅ 포함: 토큰 하드코딩 제거, WCAG AA 미달 대비 개선, 기존 화면의 컴포넌트가 M7~M10 신규 화면과 일관되도록 정돈
- ❌ 제외: 전체 미학 교체, 신규 화면 재설계 (각 Milestone에서 끝남)

### 의존성
- **M9, M10 필요**: Phase 3 기능이 안정되어야 정리 대상 범위가 확정됨

### 산출물
- [ ] 디자인 토큰 100% 적용 — `text-gray-` / `bg-gray-` / hex 색상 하드코딩 0건
- [ ] WCAG AA 색상 대비 통과 (기존 `text-neutral-400` 등 미달 부분 개선)
